# 自动内存管理：垃圾回收

## 人工内存管理的挑战

内存地址是有限资源。在程序运行过程中，如何回收空闲内存地址以便后续使用，是一个持续挑战的问题。

人工管理的负担：
- 避免多次释放
- 避免引用已释放的内存块
- 及时释放不再使用的内存块

```c
int send_request() {
    size_t n = read_size();
    int *elements = malloc(n * sizeof(int));
    if (read_elements(n, elements) < n) {
        // elements not freed!
        return -1;
    }
    // ...
    free(elements);
    return 0;
}
```

## 智能指针

智能指针通过将指针封装成局部对象，程序离开 scope 时自动释放内存。

### unique_ptr (C++)

```cpp
void uniquePtrExample() {
    std::unique_ptr<Widget> ptr1 = std::make_unique<Widget>(1);
    ptr1->process();

    // 不能复制，只能移动
    std::unique_ptr<Widget> ptr2 = std::move(ptr1);

    // 自动释放，离开作用域时自动销毁
}
```

### shared_ptr (C++)

共享指针借助引用计数自动管理，计数为 0 时自动释放。

```cpp
auto elements = std::make_shared<vector<int>>();
store_in_cache(elements);
// process elements further
return 0;
```

**缺点**：
- 需要人工选择智能指针类型
- 不能处理复杂场景（如多线程环境共享对象）

## 垃圾回收

如果对于所有对象，都借助运行时实现自动内存管理，则程序员不再需要考虑这些细节。这称为**垃圾回收**。

### 基于引用计数的垃圾回收

每个对象维护一个计数器，记录当前有多少个引用指向它。

**基本过程**：
- 对象被创建时，计数器设为 1
- 新引用指向对象时，计数器加 1
- 引用被覆盖或超出作用域时，计数器减 1
- 计数器变为 0 时，对象被立即回收

**优点**：思路简单，实时回收
**缺点**：无法处理**循环引用**

```python
a = new Object()  # 引用计数 a=1
b = new Object()  # 引用计数 b=1
a.ref = b         # b 的计数变为 2
b.ref = a         # a 的计数变为 2
# 即使外部引用消失，a 和 b 也无法被回收
```

### 基于标记-清除的垃圾回收

**基本概念**：
- **GC Roots**：程序员正在直接使用的对象（局部变量或静态变量）
- **活跃对象**：从 GC Roots 出发可以到达的对象
- **垃圾**：没有任何引用或指针可以到达的对象

**第一阶段：标记 (Mark)**

从 GC Roots 出发，沿着引用链遍历所有可达对象，在头部标记"存活"。

**第二阶段：清除 (Sweep)**

遍历堆中所有对象，未标记的对象作为"垃圾"回收，加入空闲列表。

**优点**：没有循环引用问题
**缺点**：
- **STW 问题**：标记和清除阶段通常需要暂停所有应用线程 (Stop-The-World)
- **遍历开销**：需要遍历整个存活对象图

## 各类语言的垃圾回收策略

| 语言/平台 | 回收策略 |
|-----------|---------|
| C/C++ | 手动回收：free, delete |
| C++ | 智能指针、共享指针 |
| Java | 基于标记-清除的 GC |
| Python, iOS | 基于引用计数的 GC |
| OS 内核 | 引用计数（内核对象、共享文件） |
