# 缓冲区溢出攻击

> CMU 15-213: Introduction to Computer Systems

---

## 原理

### 什么是缓冲区溢出

超过为一个数组分配的内存边界进行写入，也称为**栈破坏（stack smashing）**。

### 示例：数组越界

```c
typedef struct {
    int a[2];
    double d;        // format: 1:11:52
} struct_t;

double fun(int i) {
    volatile struct_t s;
    s.d = 3.14;
    s.a[i] = 0x40000000;    // 可能会越界
    return s.d;
}
```

| 调用 | 结果 |
|------|------|
| `fun(0)` | 3.14 |
| `fun(1)` | 3.14 |
| `fun(2)` | 3.1399998664856 |
| `fun(3)` | 2.00000061035156 |
| `fun(4)` | 3.14 |
| `fun(6)` | Segmentation fault |

> 缓冲区溢出是导致安全隐患最重要的"技术"原因。最常见的形式是在输入字符串时没有检查输入的长度，尤其是栈上的字符数组。

### 危险函数

```c
// 无法限制输入字符串长度
char *gets(char *dest) {
    int c = getchar();
    char *p = dest;
    while (c != EOF && c != '\n') {
        *p++ = c;
        c = getchar();
    }
    *p = '\0';
    return dest;
}
```

其他危险函数：`strcpy`、`strcat`、`scanf`、`fscanf`、`sscanf` 等。

### 易受攻击的代码

```c
void echo() {
    char buf[4];              // 数组太小！
    gets(buf);                // 如果输入 > 4 会怎么样？
    puts(buf);
}
```

**反汇编：**

```asm
echo:
    subq    $24, %rsp         # 分配 24 字节栈空间
    movq    %rsp, %rdi
    callq   gets
    movq    %rsp, %rdi
    callq   puts
    addq    $24, %rsp
    ret
```

### 运行时栈

**正常情况（输入 <= 24 字符）：**

```
返回地址 (8 bytes)
──────────────────
20 bytes (未使用)
────────────────── ← %rsp
buf[3] buf[2] buf[1] buf[0]
```

**溢出情形 1：缓冲溢出但未破坏返回地址（运气好）**

```
返回地址 (不变)
──────────────────
33 32 31 30 39 38 37 36 35 34
──────────────────
00 32 31 30
```

程序正常运行，返回正确结果。

**溢出情形 2：破坏了返回地址，导致 segfault**

返回地址被覆盖为无效地址 → 程序中止并报告 "Segmentation fault"

**溢出情形 3：破坏了返回地址，但程序貌似正常执行**

返回地址被修改为一个巧合的合法地址，程序"返回"到不相关的代码，执行一些计算后最终可能返回到 main。

**哪种最严重？** 情形 3 和更严重的情形——**代码注入攻击**（情形 4）。

## 安全威胁

### 代码注入攻击

黑客通过 `gets()` 传入恶意字符串：

```
┌────────────────────────────────┐
│ exploit code │ pad │ 新返回地址 │
└────────────────────────────────┘
```

当函数执行 `ret` 时，会跳转到攻击者指定的恶意代码地址。

### 缓冲区溢出漏洞导致的安全事件

| 事件 | 年份 | 说明 |
|------|:----:|------|
| Internet Worm | 1988 | 利用 `fingerd` 的 `gets()` 漏洞，几小时内感染 ~6000 台电脑（当时互联网的 10%） |
| IM Wars | 1999 | AOL 利用自己 AIM 客户端中的缓冲区溢出漏洞来识别和阻止微软 MSN 客户端 |
| Twilight hack on Wii | 2000s | 利用缓冲区溢出在游戏机上运行自制软件 |

### 蠕虫 vs 病毒

| 类型 | 特点 |
|:----:|:----:|
| 蠕虫 | 可独立运行，可复制并传播 |
| 病毒 | 将自身添加到其他程序，不能独立运行 |

## 防护措施

### 1. 避免溢出漏洞（程序员责任）

使用能够限制输入字符串长度的库函数：

| 危险函数 | 安全替代 |
|----------|---------|
| `gets` | `fgets(buf, size, stdin)` |
| `strcpy` | `strncpy` |
| `strcat` | `strncat` |
| `sprintf` | `snprintf` |
| `scanf("%s", ...)` | `fgets` 或 `%ns` |

### 2. 系统级保护：ASLR

**地址空间布局随机化（ASLR）**

在程序开始时，在栈上分配随机大小的空间，导致整个程序的栈偏移随机移动。黑客难以预测攻击代码的地址（缓冲区的基地址）。

每次程序运行，栈地址都会发生变化。

```
main
┌────────────────────┐
│    Application     │
├────────────────────┤
│  Random Allocation │ ← 每次运行地址不同
├────────────────────┤
│  Stack             │ ← B 的地址每次不同
└────────────────────┘
```

### 3. 系统级保护：不可执行代码段

x86-64 增加了内存区域的"可执行"权限：

- 栈区域标记为"不可执行"
- 尝试运行栈上的代码会导致失败

**MMU 页表保护**：

| 权限位 | 作用 |
|:------:|:----:|
| SUP | 管理权限 |
| READ | 只读权限 |
| WRITE | 可写权限 |
| EXEC | 可执行权限 |

### 4. 栈上的金丝雀值（Canary）

**思路**：在栈上缓冲区后面紧挨着放一个特殊的金丝雀值，在函数返回之前检测是否被破坏。

**GCC 实现**：选项 `-fstack-protector`（默认开启）

```asm
echo:
    subq    $24, %rsp
    movq    %fs:0x28, %rax      # 获取金丝雀值
    movq    %rax, 8(%rsp)       # 拷贝到栈上
    xorl    %eax, %eax
    movq    %rsp, %rdi
    callq   gets
    movq    %rsp, %rdi
    callq   puts
    movq    8(%rsp), %rax       # 读取栈上的值
    xorq    %fs:0x28, %rax      # 与原来金丝雀值比较
    je      .L6                 # If same, OK
    callq   __stack_chk_fail    # FAIL
.L6:
    addq    $24, %rsp
    ret
```

```
调用 gets() 之前的栈：
┌──────────────────────┐
│      返回地址        │
├──────────────────────┤
│   金丝雀值 (8 bytes) │
├──────────────────────┤
│   20 bytes 未使用    │
├──────────────────────┤ ← %rsp
│ buf[3] buf[2] buf[1]│  buf[0]
└──────────────────────┘
```

当输入覆盖到金丝雀值时，返回前检测到值被改变，程序终止并报告 `*** stack smashing detected ***`。

> 名称来源：煤矿工人使用金丝雀来检测有毒气体。金丝雀对气体比人类更敏感——这里的金丝雀值也起到了类似的预警作用。

### 扩展：AddressSanitizer (ASan)

ASan 在分配的内存块前后插入**红区（redzones）**，填充特定标记值。当程序访问这些红区时会被捕获并报告错误。

## ROP 攻击

### 背景

现代防护使得传统代码注入越来越困难：

- ASLR：栈地址随机化，难以预测缓冲区的地址
- 栈不可执行：不能直接在栈上执行代码

### 思路

利用机器中已有的代码，串接成一个攻击代码序列（如 `stdlib` 中的库代码）。

**代码片段（Gadget）**：以 `ret` 指令（编码 `0xc3`）结束的连续指令序列。

```
Gadget 1 code → ret → Gadget 2 code → ret → ... → Gadget n code → ret
```

### 示例

```c
long ab_plus_c(long a, long b, long c) {
    return a*b + c;
}
```

```asm
4004d0: 48 0f af fe    imul %rsi,%rdi
4004d4: 48 8d 04 17    lea  (%rdi,%rdx,1),%rax   # 可作为 Gadget
4004d8: c3             retq
```

更灵活的方式——利用字节码中的任意位置：

```c
void setval(unsigned *p) {
    *p = 3347663060u;
}
```

```asm
4004d9: c7 07 d4 48 89 c7    movl $0xc78948d4,(%rdi)
4004df: c3                   retq

# 从 0x4004dc 开始解释：
4004dc: 48 89 c7             movq %rax, %rdi  # 另一个 Gadget!
```

### ROP 攻击的执行

通过 `ret` 指令进行触发。每个片段结束时的 `ret` 指令触发下一个代码片段。

```
           P's Stack Frame
           ┌──────────────────┐
           │  Gadget 1 地址   │
           ├──────────────────┤
           │  Gadget 2 地址   │
           ├──────────────────┤
     ←     │  Gadget 3 地址   │
           ├──────────────────┤ ← gets() 修改的区域
           │      pad         │
           │      pad         │
           Q's Stack Frame    │
           ┌──────────────────┤
           │      buf         │
           └──────────────────┘
```

### 局限性

| 防护措施 | 能否防御 ROP |
|----------|:-----------:|
| ASLR | 部分（代码基地址可预测） |
| 栈不可执行 | **不能**（代码在已有段中） |
| 金丝雀值 | 能 |
| 安全编程（fgets） | 能 |

## x86-64 Linux 内存布局

```
Hex Address
──────────────
00007FFFFFFFFFFF  ┌────────────────────┐
                  │      栈（Stack）    │ ← 运行时栈，默认 8MB
                  │                    │   随函数调用自动分配释放
                  ├────────────────────┤
                  │   ～ 空隙 ～        │
                  ├────────────────────┤
                  │  共享库             │
                  ├────────────────────┤
                  │  堆（Heap）         │ ← 动态分配（malloc等）
                  ├────────────────────┤
                  │  全局/静态数据      │ ← 全局变量、static 变量
                  ├────────────────────┤
                  │  代码/文本（Text）  │ ← 可执行机器指令，只读
0000000000400000  └────────────────────┘
```

### 内存分配示例

```c
int global = 0;
int useless() { return 0; }
int main() {
    void *p2, *p4;
    int local = 0;
    p2 = malloc(1L << 8);    // 256 B
    p4 = malloc(1L << 8);    // 256 B
    // ...
}
```

| 符号 | 地址范围 |
|:----:|:--------:|
| `local` | `0x7ffe4d3be87c`（栈） |
| `p4` | `0x8359d120`（堆） |
| `p2` | `0x8359d010`（堆） |
| `global` | `0x601060`（数据段） |
| `main()` | `0x40060c`（代码段） |
| `useless()` | `0x400590`（代码段） |

## 总结

| 概念 | 要点 |
|:----|:----|
| **缓冲区溢出** | 写入超出数组边界，可能破坏返回地址 |
| **代码注入** | 溢出 + 新返回地址 → 执行恶意代码 |
| **ASLR** | 栈地址随机化，提高攻击难度 |
| **NX 位** | 栈不可执行，阻止代码注入 |
| **金丝雀值** | 检测栈破坏，函数返回前校验 |
| **ROP** | 利用已有代码片段绕过 NX，通过 ret 串联 |
| **内存布局** | 代码、数据、堆、栈各段地址不同 |

**最有效的防御**：程序员编写安全的代码，使用 `fgets` 等安全函数替代 `gets` 等危险函数。
