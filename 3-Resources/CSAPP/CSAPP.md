## 核心知识点

### 1. 位与字节基础
- **位 (bit)**：0 或 1 的基本信息单元
- **字节 (byte)**：8 bits，可表示 0~255（十进制）或 00~FF（十六进制）
- 所有信息本质上都是 bit 序列

### 2. 进制表示
| 进制 | 基数 | 数字字符 |
|:-----|:----:|:---------|
| 二进制 | 2 | 0, 1 |
| 八进制 | 8 | 0~7 |
| 十进制 | 10 | 0~9 |
| 十六进制 | 16 | 0~9, A~F |

### 3. 布尔代数基础
- **与 AND**：A&B = 1 当且仅当 A=1 且 B=1
- **或 OR**：A|B = 1 当 A=1 或 B=1
- **非 NOT**：~A = 1 当 A=0
- **异或 XOR**：A^B = 1 当 A≠B

### 4. 位运算
- 移位运算：左移 <<（乘 2），右移 >>（除 2）
- 掩码操作：用 & 提取特定位
- 集合表示：用 bit 向量表示集合

### 5. 整数编码
- **无符号数**：直接二进制编码
- **补码 (Two's Complement)**：最高位为符号位，负数 = 取反 + 1
- 补码范围不对称：TMin = -2^{w-1}, TMax = 2^{w-1} - 1

### 6. 类型转换
- **相同位宽转换**：位模式不变，解释方式改变
- **符号扩展**：小类型 → 大类型时补符号位
- **截断**：大类型 → 小类型时丢弃高位

### 7. C 语言数据类型大小
| 类型 | 32位 | 64位 |
|:-----|:----:|:----:|
| char | 1 | 1 |
| short | 2 | 2 |
| int | 4 | 4 |
| long | 4 | 8 |
| float | 4 | 4 |
| double | 8 | 8 |
| pointer | 4 | 8 |

### 8. 重要概念
- **大端序 vs 小端序**：多字节数据在内存中的字节排列顺序
- 数据在内存中连续存储，指针保存起始地址
- 字符串以 null 结尾

## 核心知识点

### 1. 二进制小数
- 小数点右侧表示 2 的负幂次
- 只能精确表示 x/2^k 形式的有理数
- 0.1 在二进制中是无限循环小数 → 浮点误差的根源

### 2. IEEE 754 标准（1985）
基本公式：**v = (-1)^s × M × 2^E**

| 精度 | 总位 | s | exp | frac | Bias |
|:-----|:----:|:-:|:---:|:----:|:----:|
| 单精度 float | 32 | 1 | 8 | 23 | 127 |
| 双精度 double | 64 | 1 | 11 | 52 | 1023 |

### 3. 三种表示类型
| 类型 | exp 条件 | M | E |
|:-----|:---------|:--|:--|
| 规格化 | ≠ 000...0 且 ≠ 111...1 | 1.xxx...x | exp - Bias |
| 非规格化 | = 000...0 | 0.xxx...x | 1 - Bias |
| 特殊值 | = 111...1 | — | ∞ 或 NaN |

### 4. 偶数舍入 (Round-to-Even)
- 默认舍入模式，避免统计偏差
- 刚好一半时 → 保留位为偶数

### 5. 浮点运算特性
- **不满足结合律**：(3.14+1e10)-1e10 ≠ 3.14+(1e10-1e10)
- **不满足分配律**：1e20×(1e20-1e20) ≠ 1e20×1e20-1e20×1e20
- 这些特性给编译器和数值计算带来困难

### 6. 类型转换
- int → double：精确无损（int ≤ 53 bit）
- int → float：可能需要舍入
- double/float → int：截断小数部分，溢出时未定义行为

### 7. 特殊值
- +0 和 -0 是不同的编码
- NaN 的比较结果不确定

  - 比例因子 Scale 只能取 1, 2, 4, 8

### 3. 常用指令
- **mov**：数据传送（movb/movw/movl/movq）

## 1. 条件码 (Condition Codes)

### 1.1 四个条件码标志

x86-64 维护一组单比特条件码寄存器，由算术指令**隐式设置**（LEA 和 MOV 不设置条件码）：

| 标志 | 全称 | 含义 |
|------|------|------|
| **CF** | Carry Flag | 无符号溢出（最高位产生进位/借位） |
| **ZF** | Zero Flag | 运算结果为 0 |
| **SF** | Sign Flag | 运算结果为负（最高位为 1） |
| **OF** | Overflow Flag | 有符号溢出（补码溢出） |

### 1.2 隐式设置：算术运算

以 `addq Src, Dest`（`t = a + b`）为例：

| 条件码 | 设置条件 |
|--------|----------|
| CF | 最高位产生进位（无符号溢出） |
| ZF | `t == 0` |
| SF | `t < 0`（作为有符号数） |
| OF | `(a>0 && b>0 && t<0) \|\| (a<0 && b<0 && t>=0)` |

> `subq`、`imul` 等指令同样会设置条件码。

### 1.3 显式设置：CMP 指令

`cmpq Src2, Src1` 计算 `a - b`，**只设置条件码，不写回结果**：

| 条件码 | 条件 |
|--------|------|
| CF | 无符号借位（`a < b` 无符号视角） |
| ZF | `a == b` |
| SF | `(a - b) < 0` |
| OF | 补码溢出 |

### 1.4 显式设置：TEST 指令

`testq Src2, Src1` 计算 `a & b`，只设置 ZF 和 SF：

| 条件码 | 条件 |
|--------|------|
| ZF | `a & b == 0` |
| SF | `a & b < 0` |

典型用法：`testq %rax, %rax` 判断 `%rax` 是否为 0、是否为负数。

### 1.5 读取条件码：SETX 指令

根据条件码组合将目标寄存器的**最低字节**设为 0 或 1：

| 指令 | 条件 | 有符号含义 | 无符号含义 |
|------|------|-----------|-----------|
| `sete` / `setz` | ZF | 等于 | 等于 |
| `setne` / `setnz` | ~ZF | 不等于 | 不等于 |
| `sets` | SF | 负数 | — |
| `setns` | ~SF | 非负数 | — |
| `setg` | ~(SF^OF)&~ZF | 大于 | — |
| `setge` | ~(SF^OF) | 大于等于 | — |
| `setl` | SF^OF | 小于 | — |
| `setle` | (SF^OF)\|ZF | 小于等于 | — |
| `seta` | ~CF&~ZF | — | 高于 |
| `setb` | CF | — | 低于 |

> 使用后通常搭配 `movzbl` 将高 7 字节清零。

## 2. 跳转指令 (Jump Instructions)

### 2.1 无条件跳转与条件跳转

改变 `%rip`（程序计数器）的值。15 条基本跳转指令：

| 指令 | 跳转条件 | CMP 后含义 |
|------|----------|-----------|
| `jmp` | 无条件 | — |
| `je` / `jz` | ZF=1 | `a == b` |
| `jne` / `jnz` | ZF=0 | `a != b` |
| `js` | SF=1 | 负数 |
| `jns` | SF=0 | 非负数 |
| `jg` / `jnle` | ZF=0 且 SF=OF | 有符号 `a > b` |
| `jge` / `jnl` | SF=OF | 有符号 `a >= b` |
| `jl` / `jnge` | SF!=OF | 有符号 `a < b` |
| `jle` / `jng` | ZF=1 或 SF!=OF | 有符号 `a <= b` |
| `ja` / `jnbe` | CF=0 且 ZF=0 | 无符号 `a > b` |
| `jae` / `jnb` | CF=0 | 无符号 `a >= b` |
| `jb` / `jnae` | CF=1 | 无符号 `a < b` |
| `jbe` / `jna` | CF=1 或 ZF=1 | 无符号 `a <= b` |

### 2.2 跳转指令编码

使用 **PC 相对寻址**：目标地址 = 当前指令地址 + 指令长度 + 偏移量（编码在指令中）。这意味着代码段内的跳转不依赖绝对地址，有利于共享库。

## 3. 条件分支的实现

### 3.1 使用条件跳转（传统方式）

```c
// C 代码
long absdiff(long x, long y) {
    long result;
    if (x > y) result = x - y;
    else result = y - x;
    return result;
}
```

```asm
    cmpq    %rsi, %rdi      # 比较 x:y
    jle     .L4             # if (x <= y) goto Else
    movq    %rdi, %rax      # result = x - y
    subq    %rsi, %rax
    ret
.L4:                        # Else:
    movq    %rsi, %rax      # result = y - x
    subq    %rdi, %rax
    ret
```

**问题**：条件跳转可能导致流水线停顿（分支预测错误需要刷新流水线，约 15-20 个时钟周期）。

### 3.2 使用条件传送 (Conditional Move)

**原理**：同时计算两个分支的值，根据条件码选择其一，避免分支预测。

```asm
absdiff:
    movq    %rdi, %rax       # result = x - y
    subq    %rsi, %rax
    movq    %rsi, %rdx       # eval = y - x
    subq    %rdi, %rdx
    cmpq    %rsi, %rdi       # 比较 x:y
    cmovle  %rdx, %rax       # if <=, result = eval
    ret
```

**CMOV 指令集**：`cmove/cmovz`、`cmovne/cmovnz`、`cmovs`、`cmovns`、`cmovg`、`cmovge`、`cmovl`、`cmovle`、`cmova`、`cmovb` 等。

**不适用条件传送的场景**：
1. **计算代价高**：两个分支都需要大量计算（浪费流水线资源）
2. **安全风险**：`p ? *p : 0` —— 空指针解引用问题
3. **副作用**：`x > 0 ? x*=7 : x+=3` —— 条件传送会提前执行两个分支

## 4. 循环的汇编实现

### 4.1 Do-While 循环

最底层的循环形式，**至少执行一次循环体**：

```c
do {
    Body
} while (Test);
```

转换为：

```asm
loop:
    Body
    if (Test) goto loop;
```

示例：

```c
long pcount_do(unsigned long x) {
    long result = 0;
    do {
        result += x & 0x1;
        x >>= 1;
    } while (x);
    return result;
}
```

```asm
    movl    $0, %eax        # result = 0
.L2:                        # loop:
    movq    %rdi, %rdx
    andl    $1, %edx        # t = x & 0x1
    addq    %rdx, %rax      # result += t
    shrq    $1, %rdi        # x >>= 1
    jne     .L2             # if (x) goto loop
    ret
```

### 4.2 While 循环

**策略一：Jump-to-Middle**

```asm
    goto test;
loop:
    Body
test:
    if (Test) goto loop;
```

**策略二：Do-While 转换**（GCC -O1 以上默认）

```asm
    if (!Test) goto done;
loop:
    Body
    if (Test) goto loop;
done:
```

### 4.3 For 循环

```c
for (Init; Test; Update) Body
```

转换步骤：
1. 转换为等价的 while：`Init; while (Test) { Body; Update; }`
2. 再转换为 do-while 或 jump-to-middle 形式

### 4.4 三种循环的对照

| 循环类型 | 汇编特征 | 最少执行次数 |
|----------|---------|:----------:|
| **do-while** | 先执行 body，再判断 | 1 次 |
| **while** | 先判断再执行（jump-to-middle 或 guarded-do） | 0 次 |
| **for** | 与 while 相同，初始化在循环外 | 0 次 |

> GCC 对 `for` 和 `while` 的优化策略基本相同，汇编代码通常难以区分。

## 5. Switch 语句与跳转表

### 5.1 跳转表 (Jump Table) 原理

对于 case 值分布**密集**的 switch，编译器使用跳转表实现 **O(1)** 的分支跳转：

```asm
jmp *.L4(,%rdi,8)    # 取 .L4 数组中第 %rdi 个 8 字节条目，跳转到该地址
```

跳转表存放在只读数据段 `.section .rodata`：

```asm
.section    .rodata
.align 8
.L4:
    .quad    .L8     # case 0
    .quad    .L3     # case 1
    .quad    .L5     # case 2
    ...
```

### 5.2 完整示例

```c
long switch_eg(long x, long y, long z) {
    long w = 1;
    switch (x) {
        case 1:  w = y * z; break;
        case 2:  w = y / z;  /* Fall Through */
        case 3:  w += z; break;
        case 5:
        case 6:  w -= z; break;
        default: w = 2;
    }
    return w;
}
```

关键汇编模式：

```asm
switcher:
    cmpq $7, %rdi        # 检查 x 是否超出范围（无符号比较）
    ja    .L2             # 超出 -> default
    jmp   *.L4(, %rdi, 8) # 跳转表查找
```

### 5.3 跳转表 vs 条件分支树

| 特性 | 跳转表 | 条件分支树 (if-else) |
|------|:-----:|:------------------:|
| 执行时间 | O(1) | O(log N) |
| 空间占用 | O(N) | O(1) |
| 适合场景 | case 密集（范围小且 case 多） | case 稀疏（范围大且 case 少） |
| 流水线影响 | 无分支，不影响 | 分支可能预测失败 |

> LLVM 默认在 case 数目 >= 4 时使用跳转表。稀疏 case 编译器优化为决策树（二分查找）。

---

## 6. 条件码架构的优缺点

### 优点（历史价值）

- 设计统一、简洁，节省编码空间
- 运算与判断结合紧密

### 缺点（现代挑战）

- **隐式全局状态**：条件码是全局的，指令执行顺序影响其值
- **限制乱序执行和 ILP**：条件码引入虚假数据依赖
- **难以重命名**：产生假依赖，限制寄存器重命名技术
- **ISA 无法进化**：x86 历史包袱

> RISC-V 取消了条件码，使用 `blt x1, x2, label` 形式，每个分支指令直接在操作数上进行比较，避免隐式全局状态。

## 总结

| 控制结构 | 汇编实现方式 | 关键特点 |
|----------|-------------|---------|
| if-then-else | 条件跳转或条件移动 | CMOV 避免分支预测惩罚 |
| do-while | Body + 向后跳转 | 至少执行一次 |
| while | Jump-to-middle 或 guarded-do | 可能一次都不执行 |
| for | 先转 while，再转底层 | 初始化外提 |
| switch | 跳转表（密集）或决策树（稀疏） | O(1) 分支 |

## 1. 数组的分配与访问

### 1.1 一维数组

**分配**：`T A[L]` 在内存中分配连续的 `L * sizeof(T)` 字节。

```c
char string[12];    // 12 bytes
int val[5];         // 20 bytes (4 * 5)
double a[3];        // 24 bytes (8 * 3)
char *p[3];         // 24 bytes (8 * 3)，指针数组
```

**地址计算**：数组名 `A` 可视为指向第一个元素的常量指针 `T*`。

| 表达式 | 类型 | 值 |
|--------|------|----|
| `val[4]` | `int` | 值为 3 |
| `val` | `int*` | 基地址 `x` |
| `val + 1` | `int*` | `x + 4` |
| `&val[2]` | `int*` | `x + 8` |
| `*(val + 1)` | `int` | 值为 5 |

**汇编访问**：`%rdi` 为基地址，`%rsi` 为索引，步长编码在指令中。

```asm
movl (%rdi, %rsi, 4), %eax    # val[index]，步长 4
```

### 1.2 步长 (Stride)

步长即元素大小，决定了寻址时的缩放因子：

| 数组类型 | 元素大小 | 步长 | 汇编缩放因子 |
|----------|:-------:|:----:|:-----------:|
| `char[]` | 1 B | 1 | 无缩放 |
| `short[]` | 2 B | 2 | `(%rdi, %rsi, 2)` |
| `int[]` | 4 B | 4 | `(%rdi, %rsi, 4)` |
| `long[]` | 8 B | 8 | `(%rdi, %rsi, 8)` |
| `double[]` | 8 B | 8 | `(%rdi, %rsi, 8)` |

### 1.3 指针与数组的辨析

| 声明 | `A` 的类型 | `sizeof(A)` | `*A` 的类型 |
|------|-----------|:-----------:|-------------|
| `int A1[3]` | `int[3]` | 12 | `int` |
| `int *A2` | `int*` | 8 | `int` |
| `int A1[3][5]` | `int[3][5]` | 60 | `int[5]` |
| `int *A2[3]` | `int*[3]` | 24 | `int*` |
| `int (*A3)[3]` | `int(*)[3]` | 8 | `int[3]` |

## 2. 二维数组

### 2.1 嵌套数组 (Nested Arrays) — 行优先布局

`T A[R][C]` 以**行优先（Row-Major）** 顺序连续存储：

```
A[0][0] A[0][1] ... A[0][C-1] A[1][0] ... A[R-1][C-1]
<─── 第0行 ─────────><─── 第1行 ───> ... <─ 第R-1行 ─>
```

**元素地址计算**：`&A[i][j] = A + (i * C + j) * sizeof(T)`

**行向量访问**：`A[i]` 的地址 = `A + i * (C * K)`，其中 `K = sizeof(T)`。

```c
int *get_pgh_zip(int index) {
    return pgh[index];      // 返回第 index 行的起始地址
}
```

```asm
leaq (%rdi, %rdi, 4), %rax     # 5 * index
leaq pgh(, %rax, 4), %rax      # pgh + (20 * index)
```

**元素访问**：编译器在编译时已知内层维度 C，将 `C * sizeof(T)` 编码为指令中的常量。

```c
int get_pgh_digit(int index, int dig) {
    return pgh[index][dig];
}
```

```asm
leaq  (%rdi, %rdi, 4), %rax    # 5 * index
addl  %rax, %rsi               # 5 * index + dig
movl  pgh(, %rsi, 4), %eax     # M[pgh + 4*(5*index + dig)]
```

### 2.2 多级数组 (Multilevel Arrays)

外层数组存储的是**指针**，每个指针指向一个内层数组：

```c
int *univ[3] = {mit, cmu, ucb};
```

- `sizeof(univ) = 24`（3 个 8 字节指针）
- 内层数组可以**独立分配**，每行长度可以不同

**访问元素**：需要两次内存访问

```asm
salq    $2, %rsi                # 4 * digit
addq    univ(,%rdi,8), %rsi     # p = univ[index] + 4*digit
movl    (%rsi), %eax            # return *p
```

### 2.3 嵌套数组 vs 多级数组

| 属性 | 嵌套数组 `int pgh[4][5]` | 多级数组 `int *univ[3]` |
|------|:-----------------------:|:----------------------:|
| 内存布局 | 连续的行优先 | 指针数组 + 独立行 |
| 元素地址 | `pgh + 20*i + 4*j` | `*(univ + 8*i) + 4*j` |
| 内存访问次数 | **1 次** | **2 次**（先读指针，再读数据） |
| 灵活性 | 每行长度必须相同 | 每行长度可以不同 |
| 适用场景 | 固定大小矩阵 | 不规则结构（如字符串数组） |

### 2.4 定长数组 vs 变长数组

**定长数组**（编译时已知维度）：

```asm
salq $6, %rsi            # 64 * i
addq %rsi, %rdi          # a + 64*i
movl (%rdi, %rdx, 4), %eax  # M[a + 64*i + 4*j]
```

**变长数组**（运行时维度）：

```c
int var_ele(size_t n, int a[n][n], size_t i, size_t j) {
    return a[i][j];
}
```

```asm
imulq %rdx, %rdi         # n * i（必须使用乘法指令！）
leaq  (%rsi, %rdi, 4), %rax  # a + 4*n*i
movl  (%rax, %rcx, 4), %eax  # a + 4*n*i + 4*j
```

> 变长数组无法将步长编码为指令立即数，必须用乘法指令计算偏移。

## 3. 结构体布局

### 3.1 结构分配

结构体分配一块**连续内存**，按声明顺序分配各成员：

```c
struct rec {
    int a[4];           // offset: 0,  size: 16
    size_t i;           // offset: 16, size: 8
    struct rec *next;   // offset: 24, size: 8
};
// Total: 32 bytes
```

```
a[0..3]        i         next
│    16B     │  8B  │  8B  │
0            16     24     32
```

### 3.2 成员访问

编译器确定每个成员在结构体中的**偏移量**，机器码通过基址 + 偏移访问：

```c
int *get_ap(struct rec *r, size_t idx) {
    return &(r->a[idx]);     // r + 4*idx
}
```

## 4. 对齐与填充

### 4.1 对齐规则

x86-64 下的基本对齐规则：

| 类型 | 对齐要求 | 说明 |
|------|:-------:|------|
| `char` | 1 B | 任意地址 |
| `short` | 2 B | 地址最低 1 位为 0 |
| `int`, `float` | 4 B | 地址最低 2 位为 00 |
| `double`, `long`, `char*` | 8 B | 地址最低 3 位为 000 |
| `long double` (GCC) | 16 B | 地址最低 4 位为 0000 |

**三条规则**：
1. 成员变量的类型占用 K 字节，其偏移必须是 K 的倍数
2. 结构体的对齐需求为 K_max（各成员对齐需求的最大值）
3. 结构体的总大小必须是 K_max 的倍数

### 4.2 对齐示例

```c
struct S1 {
    char c;       // offset: 0,  size: 1
    // 3 bytes padding
    int i[2];     // offset: 4,  size: 8
    double v;     // offset: 16, size: 8
};
// Total: 24 bytes, alignment: 8
```

```
c │ padding │  i[0..1]  │     v     │
1B    3B       8B          8B
0    4        8          16        24
```

### 4.3 字段重排优化

通过重新排列成员顺序减少填充字节。**将大的成员放前面**：

```c
// 不好的顺序：24 bytes
struct S4 {
    char c;       // +1
    // +3 padding
    int i;        // +4
    char d;       // +1
    // +3 padding
};
// sizeof(S4) = 12 + 填充到 4 的倍数 => 实际上是 12 字节不对，int i 偏移为 4，最后 padding 到 4 的倍数 = 12 字节
// 但 char c + padding(3) + int i + char d + padding(3) = 1+3+4+1+3 = 12

// 好的顺序：8 bytes
struct S5 {
    int i;        // +4
    char c;       // +1
    char d;       // +1
    // +2 padding
};
// sizeof(S5) = 4+1+1+2 = 8
```

> **经验法则**：将成员按对齐要求**从大到小**排列（double → int → short → char），最小化填充。

### 4.4 为什么需要对齐

- CPU 内存访问以字长（8 字节）为基本单位
- **不对齐**的变量可能需要**两次内存访问**
- 编译器自动插入填充（padding）确保对齐

---

## 5. 结构体数组

数组的每个元素都需满足对齐需求：

```c
struct S2 {
    double v;     // 8 bytes
    int i[2];     // 8 bytes
    char c;       // 1 byte
    // 7 bytes padding
};
struct S2 a[10];  // 每个元素 24 字节
```

数组步长 = `sizeof(struct)`（含填充），确保每个元素的对齐。

---

## 6. 联合体 (Union)

由**最大成员**决定大小，所有成员共享同一块内存：

```c
union U1 {
    char c;
    int i[2];
    double v;
};
// Total: 8 bytes（由 double v 决定）
```

常见用途：位串重新解释（type punning）：

```c
typedef union {
    float f;
    unsigned u;
} bit_float_t;

float bit2float(unsigned u) {
    bit_float_t arg;
    arg.u = u;
    return arg.f;  // 保留位串不变，重新解释为 float
}
```

## 7. 字节序 (Endianness)

### 7.1 Little-Endian (x86-64)

**低地址存放最低有效字节**：

```c
int val = 0x15213;   // 十六进制: 0x00015213
```

内存布局（小端）：

```
地址递增 → 13 52 01 00
```

### 7.2 Big-Endian

**低地址存放最高有效字节**：

```
地址递增 → 00 01 52 13
```

### 7.3 小端序的特点

| 方面 | 说明 |
|------|------|
| **人类阅读** | 不便（字节顺序"颠倒"） |
| **类型转换** | 窄化转换（如 `int*` → `char*`）不需要移动指针 |
| **地址运算** | `(int*)p` 和 `p` 指向同一地址，低字节自然对齐 |

> 网络协议通常使用大端（网络字节序），C 提供 `htonl`（主机→网络）和 `ntohl`（网络→主机）进行转换。

## 总结

| 数据类型 | 关键点 |
|---------|--------|
| **一维数组** | 连续内存，步长 = 元素大小，编码在指令缩放因子中 |
| **嵌套数组** | 行优先连续存储，1 次内存访问，编译时固定步长 |
| **多级数组** | 指针数组间接访问，2 次内存访问，每行可不等长 |
| **结构体** | 连续内存 + 填充，成员通过偏移量访问 |
| **对齐** | 地址必须为类型大小的倍数，从大到小排列字段可减小体积 |
| **联合体** | 共享内存，由最大成员决定大小 |
| **字节序** | x86-64 小端，低地址存低位字节；网络协议使用大端 |

| 机制 | 说明 | 实现方式 |

| **传递控制** | 跳转到被调用函数并返回 | `call` / `ret` 指令 |

## 2. x86-64 运行时栈

### 2.1 栈的基本特征

- **向低地址增长**（栈顶地址 < 栈底地址）
- `%rsp` 寄存器始终指向**栈顶元素**（最低地址）
- 栈遵循 **LIFO**（后进先出）原则

```
高地址           ┌──────────────┐ ← 栈底（高地址，早期调用者的帧）
                 │              │
                 │     ...      │
                 │              │
                 ├──────────────┤
                 │  栈帧数据    │
                 ├──────────────┤
低地址           │              │ ← %rsp (栈顶，最低地址)
                 └──────────────┘
```

### 2.2 pushq / popq

```asm
pushq Rsrc
subq $8, %rsp       # 栈指针减 8
movq Rsrc, (%rsp)   # 将值写入新栈顶

popq Rdest
movq (%rsp), Rdest  # 从栈顶读值
addq $8, %rsp       # 栈指针加 8
```

## 3. call 和 ret 指令

### 3.1 call 指令

```asm
call Label
pushq %rip      # 将返回地址（call 下一条指令的地址）压栈
jmp Label       # 跳转到目标函数
```

### 3.2 ret 指令

```asm
ret
popq %rip       # 从栈顶弹出返回地址，跳转到该地址
```

### 3.3 控制流示例

```
地址      指令

400541:  mov %rdx, %rbx
400544:  callq 400550 <mult2>     # 将 0x400549 入栈
400549:  mov %rax, (%rbx)         # ← 返回地址（call 的下一条指令）
...

400550:  mov %rdi, %rax            # mult2 函数体
400553:  imul %rsi, %rax
400557:  retq                      # 弹出 0x400549，跳回
```

## 4. 参数传递约定

### 4.1 前 6 个参数（寄存器传递）

x86-64 Linux 调用约定（System V AMD64 ABI）：

| 参数序号 | 寄存器 | 说明 |
|:-------:|:------:|------|
| 1st | `%rdi` | 第一个整数/指针参数 |
| 2nd | `%rsi` | 第二个整数/指针参数 |
| 3rd | `%rdx` | 第三个整数/指针参数 |
| 4th | `%rcx` | 第四个整数/指针参数 |
| 5th | `%r8` | 第五个整数/指针参数 |
| 6th | `%r9` | 第六个整数/指针参数 |
| 返回值 | `%rax` | 函数返回值 |

> 浮点参数使用 XMM 寄存器（`%xmm0` ~ `%xmm7`），另有独立约定。

### 4.2 第 7 个及之后的参数（栈传递）

- 参数 7、8... 按**从右到左**的顺序压入**调用者的栈帧**
- 超过 6 个参数时才使用栈空间

## 5. 栈帧结构

### 5.1 栈帧布局

```
Caller Frame        ┌──────────────────┐
                    │  Arguments 7+    │ ← 由调用者准备
                    ├──────────────────┤
                    │  返回地址        │ ← call 指令压入
Callee Frame        ├──────────────────┤
                    │  Saved %rbp      │ ← 可选（帧指针）
                    ├──────────────────┤
                    │  局部变量        │
                    ├──────────────────┤
                    │  保存的寄存器    │ ← callee-saved
                    └──────────────────┘ ← %rsp
```

### 5.2 何时使用栈帧

- 寄存器不够用时
- 参数数目 > 6 个
- 局部变量过多
- 需要对局部变量取地址（`&x`）
- 局部变量是数组或结构体类型

### 5.3 示例：incr 调用

```c
long incr(long *p, long val) {
    long x = *p;
    long y = x + val;
    *p = y;
    return x;
}

long call_incr() {
    long v1 = 15213;
    long v2 = incr(&v1, 3000);
    return v1 + v2;
}
```

```asm
call_incr:
    subq    $16, %rsp           # 分配 16 字节栈空间
    movq    $15213, 8(%rsp)     # v1 = 15213
    movl    $3000, %esi         # 2nd arg = 3000
    leaq    8(%rsp), %rdi       # 1st arg = &v1
    call    incr
    addq    8(%rsp), %rax       # v1 + v2
    addq    $16, %rsp           # 释放栈空间
    ret
```

## 6. 寄存器保护约定

### 6.1 问题描述

函数 `yoo` 调用 `who` 时，`who` 可能覆盖 `yoo` 正在使用的寄存器。

### 6.2 两类寄存器

| 类型 | 寄存器 | 责任方 | 说明 |
|:----:|:------|:-----:|------|
| **Caller-saved** | `%rax`, `%rdi`~`%r9`, `%r10`, `%r11` | 调用者 | 调用者在 `call` 前保存/恢复（如果需要） |
| **Callee-saved** | `%rbx`, `%r12`~`%r15`, `%rbp`, `%rsp` | 被调用者 | 被调用者在入口保存旧值、返回前恢复 |

### 6.3 Caller-saved 保护示例

```c
long call_incr2(long x) {
    long v1 = 15213;
    long v2 = incr(&v1, 3000);
    return x + v2;                // x 在 call 后还需要使用
}
```

```asm
call_incr2:
    pushq   %rbx                # 保存 callee-saved 寄存器（%rbx）
    subq    $16, %rsp
    movq    %rdi, %rbx          # 将 x 存放在 %rbx（callee-saved，不会被子函数覆盖）
    movq    $15213, 8(%rsp)
    movl    $3000, %esi
    leaq    8(%rsp), %rdi
    call    incr
    addq    %rbx, %rax          # x + v2
    addq    $16, %rsp
    popq    %rbx                # 恢复 callee-saved 寄存器
    ret
```

### 6.4 寄存器分工总表

| 寄存器 | 用途 | 保护责任 |
|--------|------|:--------:|
| `%rax` | 返回值 | Caller |
| `%rdi`, `%rsi`, `%rdx`, `%rcx`, `%r8`, `%r9` | 参数传递 | Caller |
| `%r10`, `%r11` | 临时数据 | Caller |
| `%rbx`, `%r12`~`%r15` | 临时/局部数据 | **Callee** |
| `%rbp` | 帧指针（可选） | Callee |
| `%rsp` | 栈指针 | 特殊（函数返回时必须恢复） |

## 7. 递归

### 7.1 递归的机器级支持

递归函数的汇编实现与非递归函数**无本质区别**——关键在于每个调用都有独立的栈帧：

```c
long pcount_r(unsigned long x) {
    if (x == 0)
        return 0;
    else
        return (x & 1) + pcount_r(x >> 1);
}
```

```asm
pcount_r:
    movl    $0, %eax            # 写返回值（x == 0 的情况）
    testq   %rdi, %rdi
    je      .L6
    pushq   %rbx                # 保护 callee-saved 寄存器
    movq    %rdi, %rbx          # %rbx = x
    andl    $1, %ebx            # %ebx = x & 1
    shrq    %rdi                # x >>= 1
    call    pcount_r            # 递归调用（创建新栈帧）
    addq    %rbx, %rax          # (x&1) + 递归返回值
    popq    %rbx                # 恢复寄存器
.L6:
    ret
```

### 7.2 递归支持的关键要素

| 要素 | 机制 |
|------|------|
| 私有栈帧 | 每次调用分配独立的栈空间 |
| 栈帧保存 | 返回地址、局部变量、保存的寄存器 |
| 寄存器保护 | callee-saved 寄存器避免层间覆盖 |
| LIFO 匹配 | 栈的 FILO 特性完美匹配调用嵌套 |

> 编译器不需要为递归做特殊处理——普通函数调用的栈帧机制天然支持递归和互递归（P 调用 Q，Q 调用 P）。

## 8. 尾调用优化 (Tail Call Optimization)

### 8.1 原理

当函数的最后操作是调用另一个函数并**直接返回其结果**时，编译器可以：

- **复用当前栈帧**（不创建新栈帧）
- 使用 `jmp` 代替 `call`（跳转到目标函数）
- 目标函数 `ret` 时直接返回到**原调用者**

```c
int returnOneOpt(void) {
    return returnOne();   // 尾调用
}
```

```asm
returnOneOpt:
    jmp returnOne          # jmp 代替 call + ret
```

### 8.2 尾调用优化的好处

| 方面 | 效果 |
|------|------|
| **栈空间** | 节省 O(N) 栈空间（递归时尤为重要） |
| **性能** | 减少 call/ret 对的开销 |
| **递归深度** | 尾递归可优化为循环，避免栈溢出 |

> 尾调用优化对**尾递归**函数尤为重要——可将递归转换为迭代，消除栈溢出风险。

| **尾调用优化** | jmp 代替 call+ret，复用栈帧，节省空间 |

## 1. 缓冲区溢出原理

### 1.1 什么是缓冲区溢出

**缓冲区溢出**指写入超过数组分配边界的字节，破坏相邻内存区域。发生在栈上的缓冲区溢出称为**栈破坏（Stack Smashing）**。

### 1.2 危险函数 — gets()

```c
char *gets(char *dest) {
    int c = getchar();
    char *p = dest;
    while (c != EOF && c != '\n') {
        *p++ = c;
        c = getchar();
    }
    *p = '\0';
    return dest;
}
```

**关键缺陷**：`gets()` **无法限制输入长度**，输入超过缓冲区大小时继续写入栈。

其他危险函数：`strcpy`、`strcat`、`scanf`、`fscanf`、`sscanf`、`sprintf`。

### 1.3 易受攻击的代码

```c
void echo() {
    char buf[4];          // 仅 4 字节缓冲区！
    gets(buf);            // 输入超过 4 字节即溢出
    puts(buf);
}
```

```asm
echo:
    subq    $24, %rsp         # 分配 24 字节栈空间
    movq    %rsp, %rdi
    callq   gets
    movq    %rsp, %rdi
    callq   puts
    addq    $24, %rsp
    ret
```

### 1.4 溢出后果的四种情形

```
栈布局（调用 echo 后）：
返回地址 (8 bytes)
──────────────────
20 bytes 空间
────────────────── ← %rsp
buf[3..0] (4 bytes)
```

| 情形 | 描述 | 结果 |
|:----:|------|------|
| 1 | 溢出但未破坏返回地址 | 程序正常运行 |
| 2 | **返回地址被覆盖为无效地址** | **Segmentation Fault** |
| 3 | 返回地址被覆盖为巧合的合法地址 | 程序执行奇怪代码后可能崩溃 |
| 4 | 返回地址被覆盖为攻击者指定的地址 | **代码注入攻击** |

### 1.5 栈越界示例

```c
typedef struct {
    int a[2];
    double d;
} struct_t;

double fun(int i) {
    volatile struct_t s;
    s.d = 3.14;
    s.a[i] = 0x40000000;     // 越界写入
    return s.d;
}
```

| 调用 | 结果 |
|------|------|
| `fun(0)` | 3.14 |
| `fun(1)` | 3.14 |
| `fun(2)` | 3.1399998664856（部分覆盖 d 的低位） |
| `fun(3)` | 2.00000061035156（覆盖 d 的高位） |
| `fun(6)` | **Segmentation Fault**（破坏返回地址） |

## 2. 攻击类型

### 2.1 代码注入攻击

攻击者构造恶意输入覆盖返回地址为攻击代码的地址：

```
┌──────────────────────────────────────┐
│  Exploit Code  │  Pad  │  新返回地址  │
└──────────────────────────────────────┘
```

函数执行 `ret` 时，跳转到攻击者的代码地址执行恶意代码。

### 2.2 历史安全事件

| **Twilight Hack** | 2000s | 利用缓冲区溢出在 Wii 游戏机上运行自制软件 |

| 蠕虫 | 可独立运行，可自行复制并传播 |
| 病毒 | 将自身附加到其他程序，不能独立运行 |

## 3. 防护措施

### 3.1 安全编程（程序员责任）

| 危险函数 | 安全替代 |
|----------|---------|
| `gets` | `fgets(buf, size, stdin)` |
| `strcpy` | `strncpy` |
| `strcat` | `strncat` |
| `sprintf` | `snprintf` |
| `scanf("%s")` | `fgets` 或 `%ns` 格式限定 |

### 3.2 ASLR（地址空间布局随机化）

**原理**：程序启动时在栈上分配随机大小的空间，使栈基地址每次运行都不同。

```
main
┌────────────────────┐
│    Application     │
├────────────────────┤
│  Random Allocation │ ← 每次运行地址不同
├────────────────────┤
│  Stack             │ ← buf 的地址每次不同
└────────────────────┘
```

**效果**：攻击者难以预测注入代码的地址。

### 3.3 不可执行代码段（NX 位）

x86-64 页表支持内存区域的执行权限控制：

| 权限位 | 作用 |
|:-----:|------|
| SUP | 超级管理员权限 |
| READ | 可读 |
| WRITE | 可写 |
| EXEC | 可执行 |

栈区域标记为**不可执行**，尝试运行栈上的代码会失败。这阻止了经典的代码注入攻击。

### 3.4 金丝雀值（Stack Canary / Stack Protector）

**思路**：在栈上缓冲区之后放置一个特殊值（金丝雀），函数返回前验证其完整性。

**GCC 选项**：`-fstack-protector`（默认开启，GCC 4.8+）

```asm
echo:
    subq    $24, %rsp
    movq    %fs:0x28, %rax      # 获取金丝雀值（来自线程本地存储）
    movq    %rax, 8(%rsp)       # 存入栈上缓冲区之后
    xorl    %eax, %eax
    movq    %rsp, %rdi
    callq   gets
    movq    %rsp, %rdi
    callq   puts
    movq    8(%rsp), %rax       # 取出栈上的值
    xorq    %fs:0x28, %rax      # 与原始金丝雀比较
    je      .L6                 # 一致则安全
    callq   __stack_chk_fail    # 不一致！报告错误
.L6:
    addq    $24, %rsp
    ret
```

**栈布局**：

```
┌──────────────────────┐
│      返回地址        │
├──────────────────────┤
│  金丝雀值 (8 bytes)  │ ← 若溢出覆盖此处
├──────────────────────┤
│   20 bytes 未使用    │
├──────────────────────┤ ← %rsp
│ buf[3..0]           │
└──────────────────────┘
```

当输入覆盖到金丝雀值时，返回前检测到值变化，程序终止并报告 `*** stack smashing detected ***`。

> **名称来源**：煤矿工人用金丝雀检测有毒气体——金丝雀对气体比人类更敏感，起到预警作用。

### 3.5 AddressSanitizer (ASan)

在分配的内存块前后插入**红区（Redzones）**，填充特殊标记值。访问红区时立即捕获并报告错误，用于开发和测试阶段。

## 4. ROP 攻击（Return-Oriented Programming）

### 4.1 背景

现代多重防护使代码注入变得困难：
- ASLR：栈地址随机化
- NX 位：栈不可执行
- Canary：检测栈破坏

### 4.2 核心思路

**利用已有代码**，不注入新代码。攻击者寻找程序中已有的代码片段（称为 **Gadget**），通过精心排列的返回地址链将它们串联起来。

**Gadget**：以 `ret` 指令（编码 `0xc3`）结尾的连续指令序列。

```
Gadget 1 code → ret → Gadget 2 code → ret → ... → Gadget n code → ret
```

### 4.3 Gadget 示例

```c
long ab_plus_c(long a, long b, long c) {
    return a*b + c;
}
```

```asm
4004d0: 48 0f af fe    imul %rsi,%rdi
4004d4: 48 8d 04 17    lea  (%rdi,%rdx,1),%rax   # 可作为 Gadget: rax = rdi + rdx
4004d8: c3             retq
```

### 4.4 利用指令边界外的字节

更灵活的方式——从指令的**中间字节**开始解释，发现意料之外的 Gadget：

```c
void setval(unsigned *p) {
    *p = 3347663060u;
}
```

```asm
4004d9: c7 07 d4 48 89 c7    movl $0xc78948d4,(%rdi)
4004df: c3                   retq

4004dc: 48 89 c7             movq %rax, %rdi    # 意外发现的 Gadget!
```

### 4.5 ROP 攻击的执行流程

通过 `ret` 指令链式触发多个 Gadget：

```
           P's Stack Frame
           ┌──────────────────┐
           │  Gadget 1 地址   │ ← 第一次 ret 跳转到 Gadget 1
           ├──────────────────┤
           │  Gadget 2 地址   │ ← Gadget 1 的 ret 跳转到 Gadget 2
           ├──────────────────┤
           │  Gadget 3 地址   │ ← Gadget 2 的 ret 跳转到 Gadget 3
           ├──────────────────┤ ← 被 gets() 修改的区域
           │      pad         │
           │      pad         │
           Q's Stack Frame    │
           ├──────────────────┤
           │      buf         │
           └──────────────────┘
```

### 4.6 各防护措施对 ROP 的效果

| 防护措施 | 能否防御 ROP | 原因 |
|----------|:-----------:|------|
| ASLR | **部分** | 代码基地址若可预测则无效 |
| 栈不可执行 (NX) | **不能** | 代码在已有可执行段中，不注入新代码 |
| 金丝雀值 | **能** | 检测栈数据是否被篡改 |
| 安全编程 (fgets) | **能** | 根本防止溢出发生 |

## 2. 面向计算的五大优化思路

### 2.1 删除无用计算

不影响函数返回值、全局变量、系统环境或其它线程的计算可以直接删除。

### 2.2 编译时完成常量计算

将编译时已知的计算提前到编译期完成，减少运行时开销。

### 2.3 消除重复计算

**公共子表达式消除**：重复出现且值不变的表达式只需计算一次。

```c
// 优化前：4 次乘法
up = val[(i-1)*n + j];
down = val[(i+1)*n + j];
left = val[i*n + j-1];
right = val[i*n + j+1];

// 优化后：1 次乘法
long inj = i*n + j;
up = val[inj - n];
down = val[inj + n];
left = val[inj - 1];
right = val[inj + 1];
```

**循环不变式外提**：循环内不变的表达式提到循环前计算。

```c
// 优化前：每次迭代都计算 n*i
void set_row(double *a, double *b, long i, long n) {
    long j;
    for (j = 0; j < n; j++)
        a[n*i + j] = b[j];
}

// 优化后：n*i 只计算一次
void set_row(double *a, double *b, long i, long n) {
    long j;
    int ni = n*i;
    for (j = 0; j < n; j++)
        a[ni + j] = b[j];
}
```

### 2.4 简化复杂计算

用更快的等价表达式替代：`x * 2^k` → `x << k`（强度削弱）。

循环中用增量加法替代乘法：

```c
// 优化前：内部循环使用乘法
for (i = 0; i < n; i++) {
    int ni = n*i;
    for (j = 0; j < n; j++)
        a[ni + j] = b[j];
}

// 优化后：增量加法替代乘法
int ni = 0;
for (i = 0; i < n; i++) {
    for (j = 0; j < n; j++)
        a[ni + j] = b[j];
    ni += n;
}
```

### 2.5 减少控制开销（降低管理税）

控制结构（循环、函数调用、多态）会引入管理开销：

**递归 vs 循环**：

```c
// 递归：函数体需要 11 条指令
long fact(long n) {
    if (n < 2) return 1;
    else return n * fact(n-1);
}

// 循环：循环体仅 3 条指令
long factLoop(long n) {
    long acc = 1;
    for (long i = 2; i <= n; ++i)
        acc *= i;
    return acc;
}
```

**循环展开**：减少循环条件判断次数。

**循环融合**：将两个相邻循环合并，降低管理税的比例：

```c
// 融合前：2 * N 次管理税
for (int i = 0; i < N; i++) a[i].x = b[i].x;
for (int i = 0; i < N; i++) a[i].y = b[i].y;

// 融合后：N 次管理税
for (int i = 0; i < N; i++) {
    a[i].x = b[i].x;
    a[i].y = b[i].y;
}
```

**JIT 编译**：解释执行的管理税可以通过 JIT（即时编译）将热点代码编译为机器码来优化。

## 3. 编译器优化技术

### 3.1 编译器优化的分类

| 类别 | 技术 | 说明 |
|:----:|------|------|
| **机器无关** | 死代码消除 | 删除不影响结果的代码 |
| | 常量折叠/传播 | 编译时完成常量计算 |
| | 公共子表达式消除 | 复用相同表达式结果 |
| | 强度削弱 | 用廉价指令替换昂贵指令 |
| | 循环不变式外提 | 将循环不变计算提到循环外 |
| **机器相关** | 寄存器分配 | 将变量映射到寄存器 |
| | 指令选择 | 选择合适的机器指令 |
| | 指令调度 | 重排指令以利用流水线 |
| | 缓存局部性优化 | 数据布局优化 |

### 3.2 数据流分析 (Data-Flow Analysis)

编译器基于函数的控制流图（CFG）进行静态分析：

1. **每条语句**：分析其读写了哪些变量
2. **顺序结构**：s1 修改了 x，s2 也修改了 x，则 s1 的效果被覆盖
3. **分支结构**：考虑所有前驱路径上的修改都可能生效
4. **循环结构**：需要迭代求解（可能多次执行）

## 4. 编译器优化的局限 — 优化拦路虎

### 4.1 内存别名 (Memory Aliasing)

两个不同的指针可能指向同一内存地址，编译器无法确定它们是否独立。

```c
// sum_rows1：每次迭代都写回内存（3 次访存）
void sum_rows1(double *a, double *b, long n) {
    long i, j;
    for (i = 0; i < n; i++) {
        b[i] = 0;
        for (j = 0; j < n; j++)
            b[i] += a[i*n + j];
    }
}
```

问题：编译器怀疑 `a` 和 `b` 可能指向同一区域（即 `a == b` 或重叠），因此**必须在每次循环迭代中写回 `b[i]`**。如果 `a` 和 `b` 重叠，将 `b[i]` 保存在局部变量中会导致错误结果。

```c
// sum_rows2：使用局部变量累计（1 次访存）
void sum_rows2(double *a, double *b, long n) {
    long i, j;
    for (i = 0; i < n; i++) {
        double val = 0;          // 局部变量，
        for (j = 0; j < n; j++)  // 编译器知道局部变量与任何指针不别名
            val += a[i*n + j];
        b[i] = val;              // 最后一次性写回
    }
}
```

**解决方案**：
- 使用局部变量累计中间结果
- 使用 `__restrict` 关键字告诉编译器指针没有别名

### 4.2 函数调用

编译器难以确定函数是否有副作用，遇到函数调用通常做保守分析。

```c
// lower1：平方级复杂度 O(n²)
void lower1(char *s) {
    size_t i;
    for (i = 0; i < strlen(s); i++)    // strlen 在每次迭代都被调用！
        if (s[i] >= 'A' && s[i] <= 'Z')
            s[i] -= ('A' - 'a');
}
```

问题：编译器无法确定 `strlen` 是否每次返回相同值（因为循环体中修改了 `s` 的内容，`strlen` 可能变化），因此不能自动将其提到循环外。

```c
// lower2：线性级复杂度 O(n)
void lower2(char *s) {
    size_t i;
    size_t len = strlen(s);            // 手动将 strlen 提到循环外
    for (i = 0; i < len; i++)
        if (s[i] >= 'A' && s[i] <= 'Z')
            s[i] -= ('A' - 'a');
}
```

**解决方案**：
- 函数内联（`inline` 关键字或编译器自动内联）
- 程序员手动将不依赖循环变量的函数调用提到循环外

### 4.3 两条拦路虎的对比

| 拦路虎 | 表现 | 示例 | 解决方案 |
|:------:|------|------|---------|
| **内存别名** | 额外的内存读写 | `b[i] += a[i]` 每次都要写回 | 使用局部变量累计 |
| **函数调用** | 无法外提或内联 | `strlen` 在循环中 | 手动提到循环外 |

## 5. 两段式优化工作流

```
性能分析 → 识别瓶颈 → 应用优化 → 验证效果 → 重复
```

**推荐层级**：
1. **算法级**：选择更优的算法和数据结构（最大收益）
2. **代码级**：消除优化拦路虎，减少不必要的计算
3. **微架构级**：利用指令级并行、向量化、缓存优化

### 重要原则

> "过早优化是万恶之源" —— Donald Knuth

- 95% 的情况下不应操心微优化
- 只对关键的 3% 进行精细优化
- 先用 profiler 识别热点，再针对性优化

## 1. 性能度量：CPE

### 1.1 CPE 定义

**CPE（Cycles Per Element）** 是衡量向量/列表运算性能的标准指标：

```
T = CPE * n + Overhead
```

- `T`：总执行时间（时钟周期数）
- `n`：元素数量
- `CPE`：每个元素的处理代价（斜率）
- `Overhead`：固定开销（截距）

### 1.2 性能演化示例

**combine 函数**（向量元素累积运算）各版本 CPE 对比：

| Method | Int Add | Int Mult | Double Add | Double Mult |
|--------|:-------:|:--------:|:----------:|:-----------:|
| Combine1（未优化） | 22.68 | 20.02 | 19.98 | 20.18 |
| Combine1（-O1） | 10.12 | 10.12 | 10.17 | 11.14 |
| **Combine4**（手动优化） | **1.27** | **3.01** | **3.01** | **5.01** |
| 延迟上限 | 1.00 | 3.00 | 3.00 | 5.00 |
| 吞吐量上限 | 0.50 | 1.00 | 1.00 | 0.50 |

### 1.3 Combine4 优化手段

```c
// 优化前（combine1）：每次迭代都调用函数、通过指针写回
void combine1(vec_ptr v, data_t *dest) {
    long i;
    *dest = IDENT;
    for (i = 0; i < vec_length(v); i++) {
        data_t val;
        get_vec_element(v, i, &val);
        *dest = *dest OP val;   // 函数调用 + 别名问题
    }
}

// 优化后（combine4）：消除函数调用 + 消除别名
void combine4(vec_ptr v, data_t *dest) {
    long i;
    long length = vec_length(v);      // 函数调用外提
    data_t *d = get_vec_start(v);     // 直接获取数组指针
    data_t t = IDENT;                 // 局部变量消除别名
    for (i = 0; i < length; i++)
        t = t OP d[i];                // 无函数调用，无别名
    *dest = t;                        // 最后一次写回
}
```

**关键优化**：
1. 消除 `vec_length` 函数调用（外提到循环前）
2. 消除 `get_vec_element` 函数调用（直接使用指针遍历）
3. 使用局部变量消除内存别名

## 2. 现代 CPU 微架构

### 2.1 超标量 (Superscalar)

每个时钟周期能从顺序指令流中取多条指令，并**动态调度**执行顺序。

### 2.2 流水线 (Pipeline)

将指令执行划分为多个阶段（取指、译码、执行、访存、写回），不同指令的不同阶段可以并行处理。

### 2.3 Haswell CPU 功能单元配置

| 功能单元 | 数量 | 说明 |
|---------|:---:|------|
| Load | 2 | 含地址计算 |
| Store | 1 | 含地址计算 |
| Integer | 4 | 整数运算 |
| FP Multiply | 2 | 浮点乘法 |
| FP Add | 1 | 浮点加法 |
| FP Divide | 1 | 浮点除法 |

### 2.4 关键指令的延迟与吞吐量

| 指令 | 延迟（周期） | 吞吐量（周期/发射） |
|------|:----------:|:-----------------:|
| Load/Store | 4 | 1 |
| Integer Multiply | 3 | 1 |
| Integer/Long Divide | 3-30 | 3-30（不可流水线化） |
| Single/Double FP Multiply | 5 | 1 |
| Single/Double FP Add | 3 | 1 |

### 2.5 延迟上限 vs 吞吐量上限

- **延迟上限（Latency Bound）**：由操作的数据依赖链长度决定，CPE >= 操作的延迟
- **吞吐量上限（Throughput Bound）**：由功能单元数量决定，CPE >= 1/功能单元数

| 运算 | 延迟上限 | 吞吐量上限 |
|------|:-------:|:----------:|
| Int Add | 1.00 | 0.50（4 个 Integer 单元可并行） |
| Int Mult | 3.00 | 1.00 |
| Double Add | 3.00 | 1.00 |
| Double Mult | 5.00 | 0.50（2 个 FP Multiply 单元） |

> Combine4 的整数加法 CPE = 1.27，接近延迟上限 1.00；整数乘法 CPE = 3.01，几乎等于延迟上限 3.00——说明当前瓶颈在**延迟**而非吞吐量。

## 3. 指令级并行的关键概念

### 3.1 数据依赖 (Data Dependency)

```c
t = t OP d[i];    // 每次迭代依赖于前一次迭代的结果
```

形成了一条**串行依赖链**，限制了并行度。

### 3.2 控制依赖 (Control Dependency)

循环的条件跳转需要等待前面指令完成才能确定是否跳转。分支预测错误需要十几个周期恢复。

### 3.3 指令调度

处理器动态重排指令执行顺序：

- 如果 `inst[i+1]` 依赖 `inst[i]`，则先调度不依赖的 `inst[i+k]` 与 `inst[i]` 并发执行
- 受限于：硬件计算资源总量、指令流水线资源消耗、指令间依赖关系

## 4. 循环展开优化

### 4.1 基础循环展开 (2x1)

```c
void unroll2a_combine(vec_ptr v, data_t *dest) {
    long length = vec_length(v);
    long limit = length - 1;
    data_t *d = get_vec_start(v);
    data_t x = IDENT;
    long i;
    for (i = 0; i < limit; i += 2) {
        x = (x OP d[i]) OP d[i+1];    // 仍为串行依赖
    }
    for (; i < length; i++) {
        x = x OP d[i];
    }
    *dest = x;
}
```

```c
// 依赖链：x1 = x0  OP d[0]
//          x2 = x1  OP d[1]     ← 依赖 x1
//          每次迭代仍是串行
```

| 效果 | Int Add | Int Mult | Double Add | Double Mult |
|------|:-------:|:--------:|:----------:|:-----------:|
| Combine4 | 1.27 | 3.01 | 3.01 | 5.01 |
| 2x1 展开 | **0.81** | **3.01** | **3.01** | **5.01** |

> 仅整数加法受益（减少循环管理开销），其他运算仍受延迟限制，因为依赖链长度未改变。

### 4.2 重结合循环展开 (2x1a)

```c
void unroll2aa_combine(vec_ptr v, data_t *dest) {
    long length = vec_length(v);
    long limit = length - 1;
    data_t *d = get_vec_start(v);
    data_t x = IDENT;
    long i;
    for (i = 0; i < limit; i += 2) {
        x = x OP (d[i] OP d[i+1]);    // 改变结合顺序
    }
    for (; i < length; i++) {
        x = x OP d[i];
    }
    *dest = x;
}
```

```c
// 依赖链：t1 = d[0] OP d[1]       ← 独立计算
//          x1 = x0  OP t1          ← 每两次迭代才更新一次 x
//          t2 = d[2] OP d[3]       ← 独立计算
//          x2 = x1  OP t2
//          依赖链长度减半！
```

| 效果 | Int Add | Int Mult | Double Add | Double Mult |
|------|:-------:|:--------:|:----------:|:-----------:|
| Combine4 | 1.27 | 3.01 | 3.01 | 5.01 |
| 2x1a 重结合 | 0.81 | **1.51** | **1.51** | **2.51** |

> 通过改变结合顺序打破了串行依赖链，几乎实现 **2 倍加速**！

### 4.3 独立累计器展开 (2x2)

```c
void unroll2a_combine(vec_ptr v, data_t *dest) {
    long length = vec_length(v);
    long limit = length - 1;
    data_t *d = get_vec_start(v);
    data_t x0 = IDENT;
    data_t x1 = IDENT;
    long i;
    for (i = 0; i < limit; i += 2) {
        x0 = x0 OP d[i];       // 独立累计器 1
        x1 = x1 OP d[i+1];     // 独立累计器 2
    }
    for (; i < length; i++) {
        x0 = x0 OP d[i];
    }
    *dest = x0 OP x1;
}
```

```c
// 依赖链：x0[0] → x0[2] → x0[4] → ...（每隔一个元素）
//          x1[1] → x1[3] → x1[5] → ...（每隔一个元素）
//          两条独立的依赖链并行执行！
```

| 效果 | Int Add | Int Mult | Double Add | Double Mult |
|------|:-------:|:--------:|:----------:|:-----------:|
| Combine4 | 1.27 | 3.01 | 3.01 | 5.01 |
| 2x2 独立累计器 | **0.54** | **1.01** | **1.01** | **2.51** |
| 延迟上限 | 1.00 | 3.00 | 3.00 | 5.00 |
| 吞吐量上限 | 0.50 | 1.00 | 1.00 | 0.50 |

> 2x2 展开的整数乘法 CPE = 1.01，远超延迟上限 3.00，说明两条独立依赖链让两个 FP Multiply 单元同时工作！

### 4.4 三种展开方式对比

| 方式 | 代码特征 | 依赖链 | 适用场景 |
|:----:|---------|:------:|---------|
| **2x1** | 循环步长加倍 | 1 条，长度不变 | 减少循环管理税 |
| **2x1a** | 改变结合顺序 | 1 条，长度减半 | 打破延迟限制 |
| **2x2** | 多个累计器 | **k 条**，每条长度 1/k | 利用吞吐量上限 |

## 5. 寄存器的限制：寄存器溢出

### 5.1 问题描述

过度展开消耗过多寄存器，导致部分变量必须**溢出到栈内存**，反而降低性能。

```asm
acc0 *= data[i]
vmulsd (%rdx), %xmm0, %xmm0

vmovsd 40(%rsp), %xmm0          # 从栈中读取
vmulsd (%rdx), %xmm0, %xmm0
vmovsd %xmm0, 40(%rsp)          # 写回栈
```

### 5.2 寄存器溢出的代价

- 每次溢出操作增加 **Load + Store** 开销（各 4 个周期延迟）
- 内存访问比寄存器访问慢一个数量级
- 增加指令数量，占用流水线资源

> 展开因子需要平衡：太少则无法充分利用 ILP，太多则导致寄存器溢出，性能反而下降。

## 1. SIMD 向量指令优化

### 1.1 AVX2 概述

AVX2（Advanced Vector Extensions 2）是 x86-64 的 SIMD 指令集扩展：

| 特性 | 说明 |
|------|------|
| **YMM 寄存器** | 16 个 256 位（32 字节）寄存器 |
| **数据类型** | 32 字节整数、8 个单精度浮点、4 个双精度浮点 |

### 1.2 向量化示例

```c
#include <immintrin.h>

void combine_avx2(vec_ptr v, double *dest) {
    long i;
    long size = v->len;
    double *data = v->data;
    __m256d acc = _mm256_set1_pd(1.0);    // 初始化 4 个双精度为 1.0

    for (i = 0; i + 3 < size; i += 4) {   // 每次处理 4 个元素
        __m256d vals = _mm256_loadu_pd(data + i);  // 加载 4 个双精度
        acc = _mm256_mul_pd(acc, vals);             // 4 路并行乘法
    }

    // 将向量结果归约为标量
    double tmp = 1.0;
    double res[4];
    _mm256_storeu_pd(res, acc);
    for (int k = 0; k < 4; k++) tmp *= res[k];

    // 处理剩余元素
    for (; i < size; i++) tmp *= data[i];
    *dest = tmp;
}
```

### 1.3 向量化性能

| Method | Int Add | Int Mult | Double Add | Double Mult |
|--------|:-------:|:--------:|:----------:|:-----------:|
| Scalar Best | 0.54 | 1.01 | 1.01 | 0.52 |
| **Vector Best** | **0.06** | **0.24** | **0.25** | **0.16** |
| Vec Throughput Bound | 0.06 | 0.12 | 0.25 | 0.12 |

> 向量化在标量最佳基础上再获得 4~8 倍加速，接近理论吞吐量上限。

### 1.4 SIMD 原理

**核心思想**：一条指令同时对多组数据执行相同操作（Single Instruction, Multiple Data）。

- **定制指令**：将高频操作序列融合为单一硬件指令
- **FMA（Fused Multiply-Add）**：`a * b + c` 在一条指令中完成
- **向量指令**：同时对多个数据元素执行相同运算

## 2. 线程级并行优化

### 2.1 三种并行层次

| 层次 | 粒度 | 示例 |
|:----:|:----:|------|
| **指令级并行 (ILP)** | 指令 | 流水线、超标量、循环展开 |
| **指令内并行 (SIMD)** | 数据元素 | AVX2 向量指令、GPU |
| **线程级并行 (TLP)** | 线程/进程 | 多核、超线程 |

### 2.2 超线程 (SMT)

让一个物理核心中的不同执行单元并行处理来自**两个或多个线程**的指令。不同线程的指令之间高度独立，大幅增加调度机会。

### 2.3 多核并行

硬件提供多个 CPU 核心，每个核心运行不同线程。操作系统将线程调度到不同核心上。

### 2.4 GPU 并行

- 基于 CUDA、OpenCL 的大规模并行编程
- 数百到数千个核心
- 适合数据并行任务

> 三种并行能力可以**组合使用**：多线程中每个线程使用 SIMD 指令，并充分挖掘 ILP。

## 3. 分支预测优化

### 3.1 挑战

指令控制单元需领先于执行单元取出指令。遇到条件分支时，无法确定后续指令，导致执行单元没足够的指令并行执行。

### 3.2 分支预测机制

CPU 猜测分支走向，开始执行预测位置的指令，但**不提交结果**（不修改寄存器或内存）。预测正确则提交结果，预测错误则清空流水线并重新取指。

### 3.3 分支预测错误的代价

- 现代处理器上分支预测错误需要 **15-20 个时钟周期**恢复
- 可能成为主要的性能限制因素

### 3.4 优化分支的代码

```c
// 方法 1：适合分支预测的代码（可预测数据 CPE ≈ 2.5-3.5，随机数据 CPE ≈ 13.5）
for (int i = 0; i < n; i++)
    if (data[i] >= threshold)    // 数据有序时预测成功率高
        sum += data[i];

// 方法 2：适合条件传送的代码（CPE ≈ 4.0，与数据分布无关）
for (int i = 0; i < n; i++)
    sum += (data[i] >= threshold) ? data[i] : 0;
    // 编译器可能编译为 cmov 指令

// 方法 3：查找表消除分支
int lookup[] = {0, data[i]};
sum += lookup[data[i] >= threshold];

// 方法 4：算术表达式消除分支
int mask = (-(data[i] >= threshold));  // 生成全 0 或全 1 掩码
sum += data[i] & mask;
```

| 方法 | 数据可预测 | 数据随机 | 说明 |
|:----:|:---------:|:--------:|------|
| 分支预测代码 | 2.5-3.5 | ~13.5 | 依赖数据分布 |
| 条件传送代码 | ~4.0 | ~4.0 | 与数据分布无关 |
| 查找表 | 较低 | 较低 | 需小范围值 |
| 算术表达式 | 较低 | 较低 | 无分支 |

## 5. 缓存友好编程

### 5.1 内存层次结构

现代计算机有多级缓存（L1/L2/L3）：访问速度 L1 > L2 > L3 >> 主存。缓存友好的代码充分利用**时间局部性**和**空间局部性**。

### 5.2 空间局部性：访问模式

**关键**：按行遍历（步长为 1）比按列遍历（步长大）快得多：

```c
#define N 2048

// 按列访问（步长 N）：81.8ms
void copyji(int src[N][N], int dst[N][N]) {
    int i, j;
    for (j = 0; j < N; j++)
        for (i = 0; i < N; i++)
            dst[i][j] = src[i][j];
}

// 按行访问（步长 1）：4.3ms（19 倍更快！）
void copyij(int src[N][N], int dst[N][N]) {
    int i, j;
    for (i = 0; i < N; i++)
        for (j = 0; j < N; j++)
            dst[i][j] = src[i][j];
}
```

> 按行访问步长为 1（线性访问），充分利用缓存行（通常 64 字节）的空间局部性。按列访问步长 = N * 元素大小，每次访问在不同行，破坏缓存局部性。

### 5.3 缓存优化要点

| 原则 | 说明 | 示例 |
|------|------|------|
| **时间局部性** | 访问过的数据不久后再次访问 | 循环中的累计变量 |
| **空间局部性** | 访问相邻数据 | 按行遍历数组 |
| **数据对齐** | 使数据结构对齐到缓存行边界 | 避免伪共享（False Sharing） |
| **数据分块** | 分块处理以适应缓存大小 | 矩阵分块乘法 |

## 核心知识点

### 1. 存储器层次结构概述
- **动机**：CPU 与内存之间存在巨大的速度差距（"内存墙" / Von Neumann 瓶颈）
- **应对策略**：
  1. 构造存储层次结构（利用局部性）
  2. 暂时计算别的任务（并行计算）
  3. 消除数据搬运（存内计算）
- **层次结构**：从 L0（寄存器）到 L6（远程存储），每层容量递增、速度递减、成本递减
  - L0: 寄存器（编译器管理）
  - L1/L2/L3: SRAM 缓存（硬件管理，对软件透明）
  - L4: 主存 DRAM（硬件+OS 管理）
  - L5: 本地磁盘（OS 管理）
  - L6: 远程存储（Web 服务器等）
- **核心理念**：第 k 层作为第 k+1 层的缓存，兼顾低成本（底层）和高速度（顶层）

### 2. 内存读写事务
- **读事务**：CPU 将地址放到总线 → 主存检索数据 → 数据返回 CPU 寄存器
- **写事务**：CPU 将地址和数据放到总线 → 主存写入

### 3. 局部性原理 (Principle of Locality)
- **时间局部性 (Temporal Locality)**：最近被访问过的对象很可能在不久的将来再次被访问
  - 示例：循环中的累加变量 `sum`
- **空间局部性 (Spatial Locality)**：地址相邻的对象往往在时间上被集中连续访问
  - 示例：数组的顺序遍历（步长为 1）
- **行优先 vs 列优先**：
  - C 语言按行优先（row-major order）存储多维数组
  - 行优先遍历步长为 1，空间局部性好
  - 列优先遍历步长为 N，空间局部性差
- **工作集**：程序当前"正在使用"的数据集合；局部性有助于减小工作集规模

### 4. SRAM vs DRAM

| 特性 | SRAM | DRAM |
|-----|------|------|
| 晶体管数/bit | 6 或 8 | 1 |
| 访问时间 | 约 4 ns（1x） | 约 60 ns（10x） |
| 需要刷新 | 否 | 是 |
| 需要 EDC | 可选 | 必须 |
| 成本 | 约 100x | 1x（基准） |
| 应用 | 缓存（L1/L2/L3） | 主存、帧缓冲区 |

- **DRAM 组织**：d × w DRAM，总计 d×w bits，组织为 d 个超级单元（supercells），每个 w bits
- **RAS (Row Access Strobe)**：选择行，将整行复制到行缓冲区
- **CAS (Column Access Strobe)**：选择列，将超级单元从缓冲区复制到数据线

### 5. 磁盘与 SSD
- **磁盘访问时间**：`T_access = T_avg_seek + T_avg_rotation + T_avg_transfer`
  - 示例（7200 RPM）：9 ms + 4 ms + 0.02 ms ≈ 13.02 ms
  - 比 SRAM 慢约 40,000 倍，比 DRAM 慢约 2,500 倍
- **SSD (Flash)**：
  - 以页（4-512 KB）为单位读写
  - 块在擦除后才能写入，每块约 10,000 次擦写寿命
  - 顺序访问远快于随机访问；通过磨损均衡延长寿命
  - 优点：无机械部件，更快、更低功耗、更坚固
  - 缺点：磨损失效，每字节成本更高

### 6. 缓存基本概念
- **缓存**：对软件透明（不可见），由硬件根据 load/store 操作管理
- **内存**：对软件可见，指令可直接通过内存地址寻址
- **缓存行 (Cache Line)**：缓存与内存之间数据传输的基本单位

### 7. 缓存组织方式

| 类型 | 描述 | 特点 |
|------|------|------|
| **直接映射 (Direct-Mapped)** | 每个内存块只能映射到缓存中一个特定位置 | 简单、快速，但冲突缺失多 |
| **组相联 (Set-Associative)** | 每组的行数（E-way）固定，内存块可放到组内任意行 | 平衡了成本和缺失率 |
| **全相联 (Fully-Associative)** | 所有行属于一组，内存块可放到任意位置 | 缺失率最低，但成本高、速度慢 |

### 8. 缓存映射过程
- **地址划分**：`[Tag] [Index] [Block Offset]`
- **Index**：选择缓存中的组（set）
- **Tag**：与缓存行中的 tag 比较，判断是否命中
- **Block Offset**：在缓存行内定位具体字节
- **有效位 (Valid bit)**：判断该行是否包含有效数据

### 9. 写策略

| 策略 | 描述 | 特点 |
|------|------|------|
| **通写 (Write-Through)** | 写入缓存的同时写入下一级 | 简单，但产生大量总线流量 |
| **回写 (Write-Back)** | 仅写入缓存，标记 dirty 位，驱逐时才写回 | 减少总线流量，更复杂 |

- **写分配 (Write-Allocate)**：写缺失时先将块加载到缓存再写入
- **非写分配 (No-Write-Allocate)**：写缺失时直接写入下一级，不加载到缓存

### 10. 趋势与总结
- CPU 时钟（1985: 6 MHz → 2015: 3000 MHz），约 500x 增长
- 存储容量增长远快于速度增长，性能差距持续扩大
- 精心编写的程序具有良好的局部性
- 基于缓存的存储层次结构通过利用局部性弥合性能鸿沟

## 核心知识点

### 1. 链接 (Linking) 概述
- **定义**：将多个代码和数据片段合并成一个单一文件（可执行文件）的过程
- **链接器的两大任务**：
  1. **符号解析 (Symbol Resolution)**：将每个符号引用关联到一个符号定义
  2. **重定位 (Relocation)**：合并目标文件并为符号分配运行时地址

### 2. 为什么要用链接器？
- **模块化**：
  - 程序可分解成多个小模块，解耦合，支持并行开发
  - 常用函数组织成库，便于复用
- **效率**：
  - **时间效率**：修改单个源文件只需重新编译该文件，然后重新链接
  - **空间效率**：可执行文件只包含真正使用的函数

### 3. ELF 目标文件格式
- **ELF (Executable and Linkable Format)** 是 Unix/Linux 目标文件的标准二进制格式
- **三种目标文件**：

| 类型 | 后缀 | 说明 |
|------|------|------|
| 可重定位目标文件 | .o | 可与其他文件合并成可执行文件 |
| 可执行目标文件 | (无) | 可直接拷贝到内存并执行 |
| 共享目标文件 | .so | 可加载到内存动态链接（Windows DLL） |

- **ELF 文件结构**：
  - `ELF header`：魔数、类型、入口点地址等
  - `.text`：代码段
  - `.rodata`：只读数据（跳转表、常量字符串、常量浮点数）
  - `.data`：已初始化的全局/静态变量
  - `.bss`：未初始化的全局/静态变量（不占文件空间）
  - `.symtab`：符号表
  - `.rel.text` / `.rel.data`：重定位信息
  - `.debug`：调试符号信息
  - `Section header table`：节头表

- **链接视图 vs 执行视图**：
  - **链接视图**（可重定位文件）：由节（section）组成
  - **执行视图**（可执行文件）：由段（segment）组成，描述节如何映射到内存

### 4. 链接器符号分类

| 符号类型 | 说明 | 示例 |
|----------|------|------|
| **全局符号 (Global)** | 当前模块定义，可被其他模块引用 | 非 static 函数和全局变量 |
| **外部符号 (External)** | 当前模块引用，在其他模块定义 | 引用外部函数或变量 |
| **局部符号 (Local)** | 当前模块定义，只被当前模块引用 | static 函数和全局变量 |

> 注意：局部链接器符号 ≠ 程序局部变量。局部 non-static 变量存于寄存器或栈；局部 static 变量存于 .bss 或 .data

### 5. 强符号 vs 弱符号

| 类型 | 定义 |
|------|------|
| **强符号** | 函数定义、已初始化的全局变量 |
| **弱符号** | 未初始化的全局变量 |

### 6. 符号解析三条规则
1. **规则 1**：不允许存在多个强符号（链接器报错）
2. **规则 2**：如果有 1 个强符号和多个弱符号，所有引用解析到强符号
3. **规则 3**：如果有多个弱符号，任意选择一个（**危险！** 可能导致不易发现的 bug）

### 7. 全局变量使用建议
- 尽量避免使用全局变量
- 尽量使用 `static`
- 定义全局变量请初始化
- 引用外部全局变量请使用 `extern`
- GCC 选项：`-fno-common`

### 8. 重定位过程
- **地址分配**：多文件合并，为每个节和符号分配运行时地址
- **符号重写**：借助重定位表，修正寻址字段

### 9. 重定位类型

| 类型 | 描述 | 公式 |
|------|------|------|
| **绝对寻址 (Abs32)** | 将符号的绝对地址直接写入指令 | `address = symbol_addr` |
| **PC 相对寻址** | 相对 %RIP 的偏移寻址 | `address = symbol_addr - (ref_addr + addend)` |

- **重定位项格式**：`offset`（节内偏移）+ `symbol`（符号）+ `type`（类型）+ `addend`（修正量）

### 10. 常用工具
- `objdump`：查看目标文件内容
- `readelf`：读取 ELF 文件信息
- GNU Binutils：ELF 的"瑞士军刀"

## 核心知识点

### 1. 静态库 (`.a`)
- **定义**：将多个可重定位目标文件串接成一个带索引的单一文件（archive 包）
- **创建**：`ar rs libc.a atoi.o printf.o ... random.o`
- **常见静态库**：`libc.a`（1496 个 .o 文件）、`libm.a`（444 个 .o 文件）
- **链接静态库**：`gcc -L. main2.o -lvector.a`

### 2. 链接器解析外部符号的算法
- **从左至右**单次扫描命令行参数中的文件
- 维护三个列表：
  1. 已定义符号表（E）
  2. 未定义符号表（U，待解析）
  3. 未定的弱符号表（D，备用）
- 遇到 `.a` 文件时，遍历其中每个 `.o` 文件：若能补全 U 则提取链接，否则丢弃
- **关键**：`.a` 文件应放在 `.o` 文件之后，否则可能导致 `undefined reference`
- 循环依赖需重复列出：`gcc -static -o myfunc func.o libx.a liby.a libx.a`

### 3. 静态库的缺点
1. 二进制文件中大量重复的库函数（每个程序都包含 libc 的一部分）
2. 运行时内存中存在大量重复的库函数副本
3. 修复库的 bug 需要重新链接所有应用程序

### 4. 共享库 (`.so` / `.dll` / `.dylib`)
- **优点**：
  - 可被多个进程共享（物理内存只存一份）
  - 更新库文件无需重新链接应用程序
  - 减小可执行文件体积
- **加载时链接**：程序首次加载时由动态链接器（`ld-linux.so`）自动完成
- **运行时链接**：通过 `dlopen()` / `dlsym()` / `dlclose()` 接口在程序运行中动态加载

### 5. 位置无关代码 (PIC)
- **动机**：共享库需要能被加载到任意内存地址执行
- **GCC 选项**：`-fPIC`
- **4 种引用情况**：

| 引用类型 | 解决方法 |
|----------|----------|
| 模块内代码引用 | PC 相对偏移寻址 |
| 模块内数据引用 | 运行时 PC + 偏移 |
| 模块间数据引用 | GOT（全局偏移表） |
| 模块间代码引用 | GOT + PLT（过程链接表） |

### 6. GOT (Global Offset Table)
- `.data` 节中的指针数组，指向全局变量
- GOT 与引用数据的指令之间相对距离固定
- 遵循 "Any problem in computer science can be solved with another level of indirection"

### 7. PLT (Procedure Linkage Table) 与延迟绑定
- **延迟绑定**：不在加载时重定位，而是延迟到第一次函数调用时
- **GOT 特殊项**：
  - GOT[0]：`.dynamic` 节首址
  - GOT[1]：动态链接器标识信息
  - GOT[2]：动态链接器延迟绑定代码入口
  - GOT[3+]：各共享库函数的地址（首次未解析时指向 PLT 桩）
- **首次调用流程**：PLT 桩 → 动态链接器解析地址 → 更新 GOT → 后续调用直接跳转

### 8. 共享库加载过程
- **启动时**：级联搜索和加载所有依赖的 so 文件
  1. 扫描 `.dynamic` 段识别依赖
  2. 搜索文件路径（`ld.so.cache`）
  3. 加载 so 到内存
  4. 绑定所有库变量
  5. 级联递归
- **运行时**：通过延迟绑定机制解析函数地址

### 9. 库插桩 (Library Interpositioning)
- **定义**：截获对任意函数的调用的技术
- **应用场景**：安全（沙盒/加密）、调试（跟踪）、监控与侧写（计数/跟踪）

### 10. 三种插桩方式对比

| 方式 | 时机 | 原理 | 命令示例 |
|------|------|------|----------|
| **编译时插桩** | 编译 | 宏替换，将 malloc/free 替换为自定义函数 | `#define malloc(size) mymalloc(size)` |
| **链接时插桩** | 静态链接 | 使用 `--wrap` 标志，`malloc` → `__wrap_malloc`，`__real_malloc` → `malloc` | `gcc -Wl,--wrap,malloc` |
| **加载/运行时插桩** | 动态链接 | 利用 `LD_PRELOAD` 环境变量覆盖库函数，使用 `dlsym(RTLD_NEXT, ...)` | `LD_PRELOAD="./mymalloc.so" ./prog` |

## 核心知识点

### 1. 异常控制流 (Exceptional Control Flow, ECF)
- **处理器默认行为**：读取指令 → 执行 → 准备下一条指令地址 → 重复
- **正常控制流变更**：跳转/分支、调用/返回
- **异常控制流**：响应系统状态变化的机制，发生在计算机系统各个层次

| 层次 | 机制 | 实现 |
|------|------|------|
| 低层次 | 异常 | 硬件 + 操作系统 |
| 中层次 | 进程上下文切换 | OS 软件 + 硬件定时器 |
| 中层次 | 信号 | OS 软件 |
| 高层次 | 非本地跳转 | C 运行时库 |

### 2. 异常 (Exception)
- **定义**：响应处理器状态变化时，控制流被转移到操作系统内核的过程
- **内核**：操作系统中常驻内存的部分
- **异常表 (Exception Table)**：每种事件有唯一异常号 k，作为异常表的索引（中断向量）

### 3. 异常的分类

#### 异步异常（中断）—— 来自处理器外部
- **可屏蔽中断**：定时器中断（进程调度、系统时钟维护）
- **非可屏蔽中断**：外部设备 I/O 中断（Ctrl-C、网络数据包、磁盘数据）
- **返回行为**：返回到下一条指令

#### 同步异常 —— 来自处理器内部

| 类型 | 特征 | 示例 | 返回行为 |
|------|------|------|----------|
| **陷阱 (Trap)** | 有意的 | 系统调用、断点陷阱 | 返回到下一条指令 |
| **故障 (Fault)** | 无意但可能可恢复 | 缺页异常（可恢复）、保护错误（不可恢复） | 重新执行当前指令或终止 |
| **终止 (Abort)** | 无意且不可恢复 | 非法内存地址、非法指令、奇偶校验错 | 终止当前程序 |

### 4. 系统调用
- **定义**：操作系统内核给应用程序提供的核心接口，完成用户态 → 内核态切换
- **x86-64 调用约定**：
  - `%rax`：系统调用号
  - `%rdi`, `%rsi`, `%rdx`, `%r10`, `%r8`, `%r9`：参数 1-6
  - 返回值在 `%rax`，负值表示错误
- **syscall 指令**：触发陷阱，进入内核

| 系统调用号 | 函数 | 描述 |
|-----------|------|------|
| 0 | read | 读文件 |
| 1 | write | 写文件 |
| 2 | open | 打开文件 |
| 57 | fork | 创建进程 |
| 59 | execve | 执行程序 |
| 60 | _exit | 终止进程 |
| 62 | kill | 发送信号 |

- **系统调用 vs 库函数**：
  - 系统调用：OS 内核接口，man 2
  - 库函数：语言库提供，man 3（如 `fopen` vs `open`，`printf` vs `write`）

### 5. 进程 (Process)
- **定义**：一个正在运行的程序的实例
- **两个关键抽象**：
  1. **逻辑控制流**：看似程序独占 CPU（由上下文切换提供）
  2. **私有地址空间**：看似程序独占主存（由虚拟内存提供）
- **并发 vs 并行**：
  - **并发**：两个进程执行在时间上有重叠
  - **并行**：物理上的同时发生

### 6. 上下文切换 (Context Switch)
- **内核**：常驻内存、共享的操作系统代码块，不是独立进程，而是作为现有进程的一部分运行
- **上下文切换过程**：
  1. 保存当前进程的寄存器值到 PCB
  2. 恢复新进程的寄存器值
  3. 更新页表
  4. 切换控制流
- **触发时机**：定时器中断、I/O 等待、系统调用阻塞

### 7. 关键异常示例
- **缺页异常 (Page Fault)**：访问的虚存页不在物理内存中 → 内核从磁盘加载页 → 重新执行指令
- **段错误 (Segmentation Fault)**：访问非法内存地址 → 内核向进程发送 SIGSEGV → 终止进程

## 核心知识点

### 1. 系统调用错误处理
- **规则**：必须检查每个系统级函数的返回状态
- Linux 系统级函数失败时返回 -1，设置全局变量 `errno`
- **推荐做法**：使用错误处理包装函数（如 `Fork()` 包装 `fork()`）

```c
pid_t Fork(void) {
    pid_t pid;
    if ((pid = fork()) < 0)
        unix_error("Fork error");
    return pid;
}
```

### 2. 进程状态

| 状态 | 描述 |
|------|------|
| **Running** | 进程正在执行指令 |
| **Ready** | 进程可执行但尚未被调度（等待 CPU） |
| **Blocked / Sleeping** | 进程等待外部事件（如 I/O） |
| **Stopped** | 进程被用户操作阻止执行（如 Ctrl-Z） |
| **Terminated / Zombie** | 进程已完成，但父进程尚未确认 |

### 3. 进程终止
- **三种原因**：
  1. 从 main 返回（自动调用 exit）
  2. 显式调用 `exit(int status)`
  3. 收到默认行为为终止的信号
- `exit` 被调用一次，从不返回；内核立即终止进程并通知父进程

### 4. fork() 机制
- **原型**：`pid_t fork(void);`
- **特点**：
  - 调用一次，返回两次
  - 父进程返回子进程 PID（正数）
  - 子进程返回 0
  - 出错返回 -1
- **子进程继承的内容**：
  - 用户级虚拟地址空间（独立副本，通过写时复制优化）
  - 打开文件描述符（副本，共享文件偏移）
- **关键点**：
  - 并发执行，父子进程的执行顺序不可预测
  - 两份独立但相同的地址空间
  - 共享文件 stdout
- **进程图 (Process Graph)**：刻画并发程序中语句偏序关系的工具

### 5. 进程控制块 (PCB)
- 内核为每个进程维护的进程控制块，包含：
  - PID、状态、寄存器上下文
  - 打开文件描述符表
  - 内存管理信息（页表、vm_area_struct）
  - 信号相关数据

### 6. 回收子进程：wait / waitpid
- **僵尸进程 (Zombie)**：进程已终止但尚未被父进程回收，仍占用系统资源
- **wait**：`pid_t wait(int *status);`
  - 挂起当前进程直到任意一个子进程终止
  - 返回终止子进程的 PID
- **waitpid**：`pid_t waitpid(pid_t pid, int *status, int options);`
  - 更灵活，可等待特定子进程或进程组
- **孤儿进程**：父进程先于子进程终止 → 被 `init` 进程（PID=1）收养并回收

### 7. 状态代码宏 (`sys/wait.h`)

| 宏 | 功能 |
|----|------|
| `WIFEXITED(status)` | 是否正常退出 |
| `WEXITSTATUS(status)` | 获取退出状态码 |
| `WIFSIGNALED(status)` | 是否被信号终止 |
| `WTERMSIG(status)` | 获取终止信号 |
| `WIFSTOPPED(status)` | 是否暂停 |
| `WSTOPSIG(status)` | 获取暂停信号 |

### 8. execve：创建不同进程
- **原型**：`int execve(char *filename, char *argv[], char *envp[]);`
- **特点**：调用一次，永不返回（除非出错）
- **行为**：在当前进程中加载并运行新程序
  - 覆盖：代码段、数据段、栈、堆
  - 保留：PID、打开的文件描述符、信号上下文
- **新程序启动时寄存器**：`argc` → `%rdi`，`argv` → `%rsi`，`envp` → `%rdx`
- **典型用法**：fork 后在子进程中调用 execve 加载新程序

### 9. 进程管理函数总结

| 操作 | 函数 | 特点 |
|------|------|------|
| 创建进程 | `fork` | 一次调用，两次返回 |
| 进程完成 | `exit` | 一次调用，不返回 |
| 回收等待 | `wait` / `waitpid` | 与子进程同步 |
| 加载程序 | `execve` | 一次调用，（通常）不返回 |

## 核心知识点

### 1. Linux 进程层次结构
- **PID 0**：idle 进程
- **PID 1**：init 进程（由 idle 创建），所有用户级进程的始祖（现多由 systemd 替代）
- **PID 2**：kthreadd（由 idle 创建），所有内核级进程的始祖

### 2. Shell 程序
- **定义**：代表用户运行程序的应用程序
- **常见 Shell**：sh（Bourne shell）、csh/tcsh（BSD C shell）、bash（Bourne-Again Shell）
- **基本循环**：REPL（Read-Eval-Print Loop）

### 3. Shell 中的作业管理

#### 前台进程 (Foreground) vs 后台进程 (Background)

| 特性 | 前台进程 | 后台进程 |
|------|---------|---------|
| Shell 阻塞 | 阻塞 Shell | 非阻塞 Shell |
| 信号接收 | 接收 Ctrl-C, Ctrl-Z | 忽略 Ctrl-C, Ctrl-Z |
| 标准输入 | 从键盘读取 | 无法从键盘读取 |
| 启动方式 | 直接执行命令 | 命令末尾加 `&` |

#### 作业控制命令

| 命令 | 作用 |
|------|------|
| `Ctrl-C` | 发送 SIGINT 终止前台进程 |
| `Ctrl-Z` | 发送 SIGTSTP 暂停前台进程 |
| `fg` | 将作业转前台 |
| `bg` | 将作业转后台 |
| `jobs` | 显示当前作业队列 |

### 4. Shell 的 eval 函数缺陷
- 简单 Shell 使用 `waitpid(pid, &status, 0)` 回收前台子进程
- **问题**：后台子进程终止后会变成僵尸进程（Shell 通常不会终止，因此无法自动回收）

### 5. 信号 (Signal)
- **定义**：一条小消息，通知进程系统中发生了某种类型的事件
- **信号生命周期**：由内核发送（或由其他进程请求）→ 目标进程接收 → 目标进程响应

### 6. 常见信号

| ID | 名称 | 默认行为 | 触发事件 |
|----|------|---------|---------|
| 2 | SIGINT | 终止 | Ctrl-C |
| 9 | SIGKILL | 终止 | 杀死程序（不可忽略/覆盖） |
| 11 | SIGSEGV | 终止 | 段故障 |
| 14 | SIGALRM | 终止 | 定时器到时 |
| 17 | SIGCHLD | 忽略 | 子进程停止或终止 |

### 7. 发送信号的 3 种方式
1. **内核检测到事件**直接发送（除零错误、子进程终止）
2. **用户进程请求内核发送**：`kill(pid, sig)` 或 `/bin/kill` 工具
3. **键盘**：Ctrl-C（SIGINT）、Ctrl-Z（SIGTSTP）

### 8. 进程组
- 每个进程属于一个进程组
- `getpgrp()`：返回进程组 ID
- `setpgid()`：改变进程组
- 键盘发送的信号会发送到前台进程组中的所有作业

### 9. 接收信号与响应
- **忽略**信号
- **终止**进程（可能 core dump）
- **捕获**信号：执行信号处理程序

### 10. 待处理信号与阻塞信号
- **待处理 (Pending)**：已发出但尚未被接收的信号。每种类型最多只有一个待处理信号（信号不排队）
- **阻塞 (Blocked)**：进程可阻塞接收某种信号，用 `sigprocmask` 设置
- 内核维护每个进程的 `pending` 和 `blocked` 位向量

### 11. 信号递交时机
- 内核从异常处理程序返回，准备将控制交给用户代码时
- **算法**：计算 `pnb = pending & ~blocked`
  - 若 `pnb == 0`：正常返回用户代码
  - 否则：选择最小的非零位 k，强制进程接收信号 k
- 用户定义的处理程序执行完毕后，自动返回被中断的指令

### 12. 设置信号处理程序
```c
handler_t *signal(int signum, handler_t *handler);
```
- `SIG_IGN`：忽略信号
- `SIG_DFL`：恢复默认行为
- 用户定义的处理函数地址

### 13. 信号屏蔽
```c
sigset_t mask, prev_mask;
Sigemptyset(&mask);
Sigaddset(&mask, SIGINT);
Sigprocmask(SIG_BLOCK, &mask, &prev_mask);  // 阻塞 SIGINT
// 临界区代码...
Sigprocmask(SIG_SETMASK, &prev_mask, NULL);  // 恢复
```

## CH8: 信号安全 — 可重入与非本地跳转

## 核心知识点

### 1. 信号处理程序的并发特性
- 信号处理程序是独立的逻辑流，与主程序并发运行
- 信号处理程序可被其他信号处理程序中断（嵌套信号）
- handler 与主函数、handler 之间存在嵌套并发，共享全局数据时可能被破坏

### 2. 异步信号安全 (Async-Signal-Safety)
- **定义**：函数在信号处理程序中可以安全调用，不会产生竞态条件
- **本质**：函数要么是可重入的（reentrant），要么不能被信号处理程序中断

| **安全的函数** | **不安全的函数** |
|---------------|-----------------|
| `_exit()` | `printf()` / `sprintf()` |
| `write()` | `malloc()` / `free()` |
| `wait()` / `waitpid()` | `exit()` |
| `sleep()` | 大多数标准 I/O 函数 |
| `kill()` / `getpid()` | `fprintf()` |

> `write()` 是唯一异步信号安全的输出函数；建议使用封装的 Safe I/O (SIO) 库

### 3. 安全的信号处理五大原则 (G0-G5)

#### G0: 处理程序要尽可能简单
- handler 中只记录一个全局标记，由主函数轮询响应
```c
volatile sig_atomic_t flag = 0;
void handler(int sig) { flag = 1; }
```

#### G1: 只调用异步信号安全的函数
- 在 handler 中只调用 `write()`、`_exit()`、`waitpid()` 等安全函数

#### G2: 保存和恢复 errno
- 防止其他 handler 覆盖 errno 值
```c
int olderrno = errno;
// handler code...
errno = olderrno;
```

#### G3: 阻塞所有信号保护共享全局数据结构
```c
sigset_t mask_all, prev_all;
Sigfillset(&mask_all);
Sigprocmask(SIG_BLOCK, &mask_all, &prev_all);
modify_shared_data();
Sigprocmask(SIG_SETMASK, &prev_all, NULL);
```

#### G4: 使用 `volatile` 声明全局变量
- 防止编译器优化导致变量被缓存到寄存器而非内存

#### G5: 使用 `volatile sig_atomic_t` 声明标志
- `sig_atomic_t` 保证原子读写（仅适用于单独读取或写入，不适用 `flag++`）

### 4. 可移植的信号处理：sigaction
- `signal()` 的行为在不同 Unix 系统上可能有差异
- 推荐使用基于 `sigaction` 封装的 `Signal()`：
```c
handler_t *Signal(int signum, handler_t *handler) {
    struct sigaction action, old_action;
    action.sa_handler = handler;
    sigemptyset(&action.sa_mask);
    action.sa_flags = SA_RESTART;  // 自动重启被中断的系统调用
    sigaction(signum, &action, &old_action);
    return old_action.sa_handler;
}
```

### 5. SIGCHLD 处理的关键问题
- **问题**：信号不排队，同时发送多个 SIGCHLD 只会保留一个
- **错误写法**：只在 handler 中调用一次 `wait()`
- **正确写法**：在循环中调用 `wait()` 直到返回 -1（`ECHILD`），确保回收所有子进程
```c
void child_handler2(int sig) {
    int olderrno = errno;
    pid_t pid;
    while ((pid = wait(NULL)) > 0) { ccount--; }
    if (errno != ECHILD) Sio_error("wait error");
    errno = olderrno;
}
```

### 6. 同步控制流：避免竞态条件
- **典型问题**：子进程在父进程执行 `addjob()` 之前已经终止
  - 顺序：子进程先结束 → handler 执行 `deletejob()` → 父进程执行 `addjob()` → job 永远未被删除
- **解决方案**：在 `fork()` 前阻塞 SIGCHLD，在 `addjob()` 完成后才恢复

### 7. sigsuspend：原子等待信号
- **问题**：临时解除阻塞后和挂起之前存在信号丢失的窗口期
- **解决方案**：`sigsuspend()` 原子地执行"解除阻塞 + 挂起等待"
```c
while (!pid)
    Sigsuspend(&prev);  // 原子切换，无信号丢失窗口
```
- 等效于以下操作的原子版本：`sigprocmask(SIG_SETMASK, &prev, NULL) + pause()`

### 8. 可重入函数 (Reentrant Functions)
- **定义**：函数在被中断后可以安全地再次被调用
- **要求**：
  - 不访问全局/静态数据，或仅通过原子操作访问
  - 不使用不可重入的锁
  - 不调用其他不可重入函数
- **示例**：`memcpy`、`strlen`、`write` 是可重入的；`malloc`、`printf` 不是

### 9. setjmp / longjmp：非本地跳转
- **用途**：以受控方式打破过程调用/返回规则，将控制转移到任意位置
- **setjmp**：`int setjmp(jmp_buf j);`
  - 设置返回点，保存 PC、寄存器、栈指针
  - 调用一次，返回一次或多次（初始返回 0，longjmp 后返回非零值）
- **longjmp**：`void longjmp(jmp_buf j, int i);`
  - 从对应 setjmp 位置返回，返回值为 i
  - 调用一次，从不返回
- **限制**：只能跳转到已被调用但尚未完成的函数的环境

### 10. 非本地跳转的典型应用
- **深层嵌套错误处理**：从深层函数直接跳回顶层恢复点
- **模拟 try-catch**：在 C 语言中模拟异常处理机制
- **信号处理中的重启**：用 `sigsetjmp`/`siglongjmp` 实现程序重启
- **轻量级协程**：保存和恢复 CPU 上下文
- **信号安全重启**：在 Ctrl-C 时重置状态而非终止

### 11. 并发控制总结
- **阻塞信号**：使用 `sigprocmask` 保护临界区
- **原子等待**：使用 `sigsuspend` 避免信号丢失窗口
- **非本地跳转**：使用 `setjmp`/`longjmp` 实现异常控制和重启

## 核心知识点

### 1. 虚拟存储器的三大动机
- **高效使用主存**：把 DRAM 用作部分虚拟地址空间的缓存（按需分页）
- **简化内存管理**：每个进程获得相同的统一线性地址空间
  - 简化链接（统一基址）、加载（消除重定位）、共享（同一物理页映射到多个虚拟地址空间）
- **隔离地址空间（保护）**：进程间不能互相干扰，用户程序不能访问内核信息

### 2. 虚拟寻址系统
- **MMU (Memory Management Unit)**：负责将虚拟地址 (VA) 转换为物理地址 (PA)
- **基本流程**：`CPU → VA → MMU → PA → 主存`

### 3. 地址空间
- **虚拟地址空间**：`N = 2^n` 个虚拟地址的集合 `{0, 1, ..., N-1}`
- **物理地址空间**：`M = 2^m` 个物理地址的集合 `{0, 1, ..., M-1}`
- **地址转换映射**：`MAP: V → P ∪ {∅}`
  - 数据在物理内存中：`MAP(a) = a'`（物理地址）
  - 数据不在物理内存中：`MAP(a) = ∅`（缺页故障）

### 4. VM 作为缓存的三个关键点
- **Page in cache (DRAM) ≪ Page out**：DRAM 容量远小于磁盘空间
- **巨大的未命中惩罚**：磁盘访问比 DRAM 慢约 10,000 倍，因此页缺失极少
- **全相联 (Fully Associative)**：每个虚拟页可放在任意物理页中

### 5. 虚拟地址格式
```
虚拟地址: [VPN (Virtual Page Number)] [VPO (Virtual Page Offset)]
物理地址: [PPN (Physical Page Number)] [PPO (Physical Page Offset)]
```
- VPO 与 PPO 在页大小相同的情况下位数相同
- VPN 通过页表转换为 PPN

### 6. 页表 (Page Table)
- **定义**：页表项 (PTE) 的数组，用虚拟页号 (VPN) 索引
- **特性**：每个进程独立的页表，是驻留在内存中的内核数据结构
- **PTE 内容**：
  - 页在内存中：物理页号 (PPN) + 状态位（有效位、权限位等）
  - 页不在内存中：磁盘上的位置

### 7. 页命中流程
1. CPU 发送 VA 到 MMU
2. MMU 以 VPN 为索引查询页表，得到 PPN
3. MMU 将 PPN + VPO 拼接成 PA，发送到主存
4. 主存返回数据给 CPU

### 8. 缺页故障 (Page Fault) 流程
1. MMU 查询页表，发现有效位为 0
2. MMU 触发缺页异常
3. 缺页处理程序：选择牺牲页（若 dirty 则写回磁盘）
4. 从磁盘读取新页到 DRAM，更新页表
5. 返回原指令重新执行
- **按需分页 (Demand Paging)**：仅在缺失时才将页复制到 DRAM

### 9. TLB (Translation Lookaside Buffer)
- **定义**：MMU 中的小型组相联硬件缓存，缓存 VPN → PPN 映射
- **典型参数**：16-512 项，命中延迟 0.5-1 周期，未命中代价 10-100 周期
- **失效率**：0.01% - 1%（利用页表访问的局部性）
- **TLB 命中**：消除一次或多次内存访问（指页表查询）
- **TLB 未命中**：需要从内存读取 PTE，由 MMU 硬件处理

### 10. 多级页表
- **线性页表的问题**：32 位地址、4KB 页、4B PTE → 4MB 页表；64 位完全不可行
- **多级解决方案**：类似稀疏矩阵表示，一级表常驻内存，二级表按需分配
- 常见架构：
  - 32 位（x86）：二级页表
  - RISC-V：四级页表（48 位 VA → 40 位 PA）
  - x86-64：四级页表

### 11. 内存保护
- PTE 增加权限位：READ、WRITE、EXEC、U/S（用户/超级用户）
- MMU 在每次访问时检查权限
- 用户进程违反权限 → 段错误（SIGSEGV）
- 只有 OS 可以修改页表

### 12. 工作集 (Working Set)
- **定义**：程序当前正在使用的一组活跃虚拟页
- **工作集大小 < 主存容量**：强制缺失后表现良好
- **所有工作集总大小 > 主存容量**：**抖动 (Thrashing)**，不断交换页导致性能崩溃
- 虚拟存储器高效的原因：局部性意味着工作集远小于地址空间

### 13. DRAM 缓存特性
- 大页容量（通常 4KB，有时 4MB）
- 全相联（任何虚拟页可放任意物理页）
- 回写策略（Write-Back）
- 偏好 LRU 替换算法

### 14. 总结
- **程序员视角**：每个进程拥有私有线性地址空间，受保护不被其他进程破坏
- **系统视角**：通过缓存虚存页高效使用内存，依赖局部性原理才有效


- **D**：脏位/dirty（被写入时设置）
- **G**：全局页（不驱逐出 TLB）
- **XD**：禁止取指（NX bit）

- **VIPT (Virtual Index, Physical Tag)**：用虚拟地址的 index 字段并行查找 cache，同时进行 TLB 地址转换；tag 比较用物理地址

1. 只复制 `mm_struct`、`vm_area_struct` 和页表
2. 所有 PTE 标记为 read-only

- **prot**：`PROT_READ` / `PROT_WRITE` / `PROT_EXEC`
- **flags**：`MAP_ANON`（匿名）/ `MAP_PRIVATE` / `MAP_SHARED`
- **典型用途**：
  - 文件 I/O（文件映射到内存）
  - 匿名内存分配
  - 共享内存（进程间通信）
- **示例**：用 mmap 实现零拷贝文件复制（无需用户空间缓冲区）
```c

}
```

## 核心知识点

### 1. 动态内存分配概述
- **作用**：程序员使用分配器（如 `malloc`）获取运行时虚拟内存
- **堆 (Heap)**：动态内存分配器管理的进程 VM 区域
  - 位于用户栈和共享库之间
  - 使用 `sbrk()` 系统调用扩展或收缩
- **适用场景**：仅在运行时才知道其大小的数据结构

### 2. malloc 程序包

| 函数 | 说明 |
|------|------|
| `void *malloc(size_t size)` | 分配 size 字节，16 字节对齐（x86-64），失败返回 NULL |
| `void free(void *p)` | 释放 p 指向的内存块（p 必须来自 malloc/calloc/realloc） |
| `Calloc` | 初始化为零的 malloc |
| `Realloc` | 修改已分配块的大小 |
| `Sbrk` | 扩展或收缩堆 |

### 3. 分配器的约束条件
- 应用程序可按任意顺序发出 malloc/free 请求
- free 必须对应 malloc 分配的块
- 必须立即响应 malloc（不能重排序或缓存）
- 必须从空闲内存中分配
- 必须满足对齐要求（64 位：16 字节对齐）
- 只能操作空闲内存
- **不能移动已分配的块**（不允许压缩）

### 4. 性能目标

| 指标 | 定义 |
|------|------|
| **吞吐量 (Throughput)** | 每单位时间完成的请求数量 |
| **峰值利用率 (Peak Utilization)** | 应用请求的最大有效负载总量 / 分配器使用的堆内存总量 |

### 5. 碎片 (Fragmentation)

| 类型 | 描述 |
|------|------|
| **内部碎片 (Internal)** | 有效负载小于内存块大小（开销、对齐填充、策略决定） |
| **外部碎片 (External)** | 总空闲内存足够，但没有单个连续空闲块足够大 |

### 6. 隐式空闲列表 (Implicit Free List)
- **核心思想**：用一个链表遍历所有内存块（已分配 + 空闲）
- **内存块格式**：
```
[头部: size | a(allocated)] [有效负载] [可选的填充]
```
- 利用对齐特性（低位始终为 0），低位存储已分配/空闲标志

### 7. 边界标记 (Boundary Tag) [Knuth73]
- **格式**：在空闲块的首尾都复制大小/标志信息
```
[头部: size | a] [有效负载和填充] [尾部: size | a]
```
- **作用**：允许反向遍历，实现常数时间合并

### 8. 查找空闲块的策略

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **首次适配 (First Fit)** | 从头搜索，选第一个合适的 | 简单 | 线性时间，开头易碎片 |
| **下一次适配 (Next Fit)** | 从上一次位置开始搜索 | 比首次快 | 碎片更多 |
| **最佳适配 (Best Fit)** | 选剩余字节最少的 | 碎片最少 | 通常更慢 |

### 9. 分割 (Splitting)
- 当找到的空闲块大于需求时，分割成两个块
- 第一个块标记为已分配（需求大小），第二个块标记为空闲（剩余部分）
- 避免内碎片的浪费

### 10. 合并 (Coalescing)
- **问题**：连续释放小块导致"虚假碎片"（总空间足够但无连续大块）

| 策略 | 描述 |
|------|------|
| **立即合并 (Immediate Coalescing)** | 每次 free 时立即合并相邻空闲块 |
| **推迟合并 (Deferred Coalescing)** | 合并操作推迟到分配失败时进行，提高分配延迟稳定性 |

- **边界标记实现常数时间合并的 4 种场景**：

| 前一块 | 后一块 | 合并操作 |
|--------|--------|----------|
| 已分配 | 已分配 | 不合并 |
| 已分配 | 空闲 | 与后合并 |
| 空闲 | 已分配 | 与前合并 |
| 空闲 | 空闲 | 三者合并 |

### 11. 堆结构维护
- 第一个块前设置**虚拟表尾**（已分配），防止合并第一个块
- 最后一个块后设置**虚拟表头**（已分配），防止合并最后一个块

### 12. 顶层 malloc 伪代码
```
1. round_up(size + header_size, alignment)
2. find_fit(asize) — 遍历空闲列表找合适块
3. 若找到：标记为已分配、分割（如需要）、返回有效负载指针
4. 若找不到：返回 NULL（或通过 sbrk 扩展堆）
```

### 13. 顶层 free 伪代码
```
1. payload → header 指针转换
2. 标记块为空闲
3. 更新头部和尾部
4. 尝试合并相邻空闲块
```

### 14. 隐式列表总结

| 维度 | 特性 |
|------|------|
| 实现复杂度 | 非常简单 |
| 分配成本 | 最坏情况线性时间 |
| 释放成本 | 最坏情况常数时间（含合并） |
| 适用场景 | 教学、简单嵌入式系统 |

## 核心知识点

### 1. 跟踪空闲块的四种方法

| 方法 | 描述 | 特点 |
|------|------|------|
| **隐式列表** | 用长度链接所有内存块 | 实现简单，分配 O(N) |
| **显式列表** | 用指针在空闲块中建立链表 | 分配 O(空闲块数)，非 O(总块数) |
| **分离式空闲列表** | 按不同的大小类建立不同的空闲列表 | 吞吐量高，碎片少 |
| **按大小排序的块** | 使用平衡树（红-黑树），按长度排序 | 分配 O(log N) |

### 2. 显式空闲列表 (Explicit Free List)
- **结构**：只维护空闲块的双向链表，利用有效负载区域保存前向/后向指针
```
[头部: size | a] [前驱指针] [后继指针] [有效负载] [尾部: size | a]
```
- **每个空闲块额外开销**：2 个字（前驱 + 后继指针）

### 3. 显式列表的释放策略

| 策略 | 描述 | 特点 |
|------|------|------|
| **LIFO** | 插入列表开始位置 | 简单且时间恒定，碎片更多 |
| **FIFO** | 插入列表结束位置 | 简单且时间恒定，碎片更多 |
| **按地址排序** | 保持地址顺序 | 碎片化更少，需搜索 |

### 4. 显式列表的合并场景
- 与隐式列表相同的四种合并场景，但需要更新链表指针
- 插入/删除空闲块时需要维护双向链表的连接关系

### 5. 分离式空闲列表 (Segregated Free List / Seglist)
- **核心思想**：若干空闲列表，每个对应一个**大小类**
- **常见大小类**：
  - 小尺寸每个大小一个类：16, 32, 48, 64, 80, 96, 112, 128, ...
  - 之后使用 2 的幂范围：[16-32] [33-64] [65-128] ...
- **分配算法**：
  1. 搜索适合大小的空闲列表（首次适配）
  2. 找到则分割并放入适当列表
  3. 没找到则尝试下一个更大类
  4. 都没找到则用 `sbrk()` 请求额外堆内存

### 6. Seglist 的优点
- **更高的吞吐量**：2 的幂大小类达到 O(log N) 时间 vs 线性
- **更好的内存利用率**：分离式列表的首次适配 ≈ 整个堆的最佳适配效果

### 7. 常见内存错误分类

| # | 错误类型 | 示例 | 检测/预防 |
|---|---------|------|-----------|
| 1 | 间接引用错误指针 | `scanf("%d", val)` 而非 `&val` | gdb, 静态分析 |
| 2 | 读取未初始化内存 | `malloc` 后未初始化就使用 | `Calloc` 替代 |
| 3 | 写覆盖内存 | 指针大小不匹配（`int**` 分配 `sizeof(int)`） | valgrind |
| 4 | **缓冲区溢出** | `gets(s)` 输入超长字符串 | 使用 `fgets` 等边界检查函数 |
| 5 | 差一错误 | `for (i=0; i<=N; i++)` 超界 | 仔细检查循环边界 |
| 6 | 错误理解指针运算 | `p += sizeof(int)` 而非 `p++` | 理解指针算术自动缩放 |
| 7 | 引用指针而非指向对象 | `*size--` 等价于 `*(--size)` | 括号明确优先级 |
| 8 | 引用不存在的变量 | 返回局部变量地址 | 使用堆分配或传参 |
| 9 | **重复释放** | 对同一地址调用两次 `free` | free 后置 NULL |
| 10 | **悬空指针/野指针** | free 后继续使用已释放内存 | free 后指针置 NULL |
| 11 | **内存泄漏** | 分配后忘记 free | valgrind 检测 |
| 12 | 仅释放数据结构的一部分 | 只 free 了链表头 | 完整遍历释放 |

### 8. 内存调试工具

- malloc 后检查返回值是否为 NULL

- 避免重复释放（double free）

- 及时释放不再使用的内存（避免内存泄漏）

### 2. 智能指针 (C++)

#### unique_ptr
- 独占所有权，不能复制只能移动
- 离开作用域自动释放资源

```cpp
std::unique_ptr<Widget> ptr = std::make_unique<Widget>(1);

// 自动释放
```

#### shared_ptr
- 共享所有权，基于引用计数
- 计数为 0 时自动释放
```cpp
auto elements = std::make_shared<vector<int>>();
```
- **缺点**：需要人工选择类型、无法处理复杂场景（如多线程环引用）

### 3. 垃圾回收 (GC) 概述
- **定义**：借助运行时实现自动内存管理，程序员无需手动释放
- **核心问题**：判断哪些对象是"垃圾"（无法再被程序到达的对象）
- **GC Roots**：程序员正在直接使用的对象（局部变量、静态变量、全局变量）
- **活跃对象**：从 GC Roots 出发可达的对象
- **垃圾**：没有任何引用或指针可以到达的对象

### 4. 引用计数 (Reference Counting)

| 特性 | 描述 |
|------|------|
| **原理** | 每个对象维护计数器，记录引用数 |
| **递增** | 新引用指向对象时 +1 |
| **递减** | 引用被覆盖或超出作用域时 -1 |
| **回收时机** | 计数器变为 0 时立即回收 |

- **优点**：思路简单，实时回收
- **缺点**：**无法处理循环引用**
```python
a = Object()  # a.refcount = 1
b = Object()  # b.refcount = 1
a.ref = b     # b.refcount = 2
b.ref = a     # a.refcount = 2
```
- **适用**：Python、iOS (ARC)、OS 内核对象引用

### 5. 标记-清除 (Mark-and-Sweep)

| 阶段 | 描述 |
|------|------|
| **标记 (Mark)** | 从 GC Roots 出发，沿引用链遍历所有可达对象，在头部标记"存活" |
| **清除 (Sweep)** | 遍历堆所有对象，未标记的作为垃圾回收，加入空闲列表 |

- **优点**：没有循环引用问题
- **缺点**：
  - **STW (Stop-The-World)**：标记和清除阶段通常需要暂停所有应用线程
  - **遍历开销**：需要遍历整个存活对象图
- **适用**：Java 等主流 GC 的基础算法

### 6. 复制收集 (Copying Collection)
- **原理**：将堆分为两个半区（from-space / to-space）
- 从 GC Roots 出发遍历存活对象，复制到另一半区
- 原始半区整体回收
- **优点**：无碎片化，分配快速（指针碰撞）
- **缺点**：内存利用率只有 50%，存活对象多时复制成本高

### 7. 分代收集 (Generational Collection)
- **经验观察**：大部分对象存活时间很短（"朝生夕灭"）
- **分代假设**：
  - **新生代 (Young Generation)**：新创建对象，频繁 GC
  - **老年代 (Old Generation)**：存活多次后晋升，较少 GC
- **优点**：针对不同代使用不同算法，显著提高整体效率
- **应用**：Java HotSpot VM、.NET CLR

### 8. 各语言/平台的垃圾回收策略

| 语言/平台 | GC 策略 |
|-----------|---------|
| C/C++ | 手动回收：`free()` / `delete` |
| C++ | 智能指针（unique_ptr, shared_ptr） |
| Java | 基于标记-清除 + 分代收集 |
| Python, iOS | 基于引用计数 + 辅助标记清除（处理环） |
| Go | 并发标记清除 |
| Rust | 所有权系统（编译时检查，无运行时 GC） |

### 9. 保守式 GC vs 准确式 GC

| 类型 | 描述 | 适用 |
|------|------|------|
| **保守式 GC** | 无法区分指针和非指针（C 语言的困境） | C/C++ |
| **准确式 GC** | 运行时/编译器能精确识别指针 | Java, Go, C# |

- C 语言中实现 GC 的困难：无法确定内存中的某个值是指针还是整数
- 保守式 GC 策略：假设所有"看起来像指针"的值都作为指针处理（可能导致误报但安全）

### 10. 内存管理策略总结

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 手动管理 | 完全控制、无运行时开销 | 易出错、开发负担重 | 系统编程、嵌入式 |
| 智能指针 | 自动管理、RAII | 仅限 C++、循环引用问题 | 现代 C++ 项目 |
| 引用计数 | 实时回收、简单 | 循环引用、计数开销 | Python, iOS |
| 标记-清除 | 无环问题 | STW 暂停、遍历开销 | Java 等基础算法 |
| 分代收集 | 高效、短暂停 | 实现复杂 | 主流工业级 VM |

## 二、条件码 (Condition Codes)

### 2.1 四个标志位的含义

| 标志 | 全称 | 含义 |
|------|------|------|
| **CF** | Carry Flag | 无符号运算溢出，最高位产生进位 |
| **ZF** | Zero Flag | 运算结果为零 |
| **SF** | Sign Flag | 运算结果为负（有符号视角的最高位） |
| **OF** | Overflow Flag | 有符号运算溢出：两个输入符号相同但输出符号不同 |

### 2.2 条件码的设置方式

**隐式设置**：大多数算术指令都会设置条件码（**但 `MOV` 和 `LEA` 不设置条件码**）。

以 `addq Src, Dest`（即 `t = a + b`）为例：

| 条件码 | 设置条件 |
|--------|---------|
| CF | 最高位产生进位（无符号溢出） |
| ZF | `t == 0` |
| SF | `t < 0`（有符号） |
| OF | `(a>0 && b>0 && t<0) \|\| (a<0 && b<0 && t>=0)` |

**显式设置 — CMP 指令**：
- `cmpq Src2, Src1` 计算 `Src1 - Src2`（注意操作数顺序与 AT&T 语法一致）
- 只设置条件码，**不修改目标寄存器**
- 用于条件判断前的比较

| CMP 后的条件码 | 含义 |
|:---|:---|
| CF = 1 | 无符号：`Src1 < Src2`（产生借位） |
| ZF = 1 | `Src1 == Src2` |
| SF = 1 | 有符号：`(Src1 - Src2) < 0` |

**显式设置 — TEST 指令**：
- `testq Src2, Src1` 计算 `Src1 & Src2`（按位与）
- 只设置 ZF 和 SF，**不修改目标寄存器**
- 常用于检测寄存器是否为 0 或某位是否为 1

| TEST 后的条件码 | 含义 |
|:---|:---|
| ZF = 1 | `Src1 & Src2 == 0` |
| SF = 1 | `Src1 & Src2 < 0`（最高位为 1） |

### 2.3 SetX 指令：条件码 → 字节

根据条件码的组合，将目标寄存器的**最低字节**设为 0 或 1，其余 7 个字节不变。通常配合 `movzbl` 清零高位。

| 指令 | 条件 | CMP 后的语义 |
|------|:----:|:------------|
| `sete` / `setz` | ZF | 相等 |
| `setne` / `setnz` | ~ZF | 不相等 |
| `sets` | SF | 负数 |
| `setns` | ~SF | 非负数 |
| `setg` | ~(SF^OF)&~ZF | 有符号大于 |
| `setge` | ~(SF^OF) | 有符号大于等于 |
| `setl` | SF^OF | 有符号小于 |
| `setle` | (SF^OF)\|ZF | 有符号小于等于 |
| `seta` | ~CF&~ZF | 无符号高于 |
| `setae` | ~CF | 无符号高于等于 |
| `setb` | CF | 无符号低于 |
| `setbe` | CF\|ZF | 无符号低于等于 |

## 三、跳转指令 (Jump Instructions)

跳转指令通过改变 `%rip` 来改变控制流。目标地址通过 **PC 相对寻址** 计算：目标地址 = 当前指令地址 + 指令长度 + 偏移量。

### 3.1 完整 15 条跳转指令表

| 指令 | 别名 | 跳转条件 | CMP 后含义 |
|:----:|:----:|:--------:|:----------|
| `jmp` | — | 无条件 | — |
| `js` | — | SF = 1 | 负数 |
| `jns` | — | SF = 0 | 非负 |
| `jo` | — | OF = 1 | 有符号溢出 |
| `jno` | — | OF = 0 | 无有符号溢出 |
| `je` | `jz` | ZF = 1 | 相等 |
| `jne` | `jnz` | ZF = 0 | 不相等 |
| `jb` | `jc, jnae` | CF = 1 | 无符号低于 |
| `jae` | `jnc, jnb` | CF = 0 | 无符号高于等于 |
| `ja` | `jnbe` | CF=0 且 ZF=0 | 无符号高于 |
| `jbe` | `jna` | CF=1 或 ZF=1 | 无符号低于等于 |
| `jl` | `jnge` | SF != OF | 有符号小于 |
| `jge` | `jnl` | SF = OF | 有符号大于等于 |
| `jg` | `jnle` | ZF=0 且 SF=OF | 有符号大于 |
| `jle` | `jng` | ZF=1 或 SF!=OF | 有符号小于等于 |

> `objdump` 和 `gdb` 反汇编时，别名指令统一显示为"名称"列的形式。

### 3.2 条件分支：if-else 的汇编实现

C 语言代码：
```c
long absdiff(long x, long y) {
    long result;
    if (x > y)
        result = x - y;
    else
        result = y - x;
    return result;
}
```

转换为 goto 形式：
```c
long absdiff_j(long x, long y) {
    long result;
    int ntest = x <= y;
    if (ntest) goto Else;
    result = x - y;
    goto Done;
Else:
    result = y - x;
Done:
    return result;
}
```

汇编代码：
```asm
absdiff:
    cmpq    %rsi, %rdi      # x:y
    jle     .L4
    movq    %rdi, %rax
    subq    %rsi, %rax
    ret
.L4:                        # x <= y
    movq    %rsi, %rax
    subq    %rdi, %rax
    ret
```

## 四、条件传送指令 (Conditional Move)

### 4.1 基本原理

条件移动指令（CMOV）根据条件码决定是否执行数据移动，**不产生分支**。

```asm
absdiff_cmov:
    movq    %rdi, %rax       # result = x
    subq    %rsi, %rax       # result = x - y
    movq    %rsi, %rdx
    subq    %rdi, %rdx       # eval = y - x
    cmpq    %rsi, %rdi       # x:y
    cmovle  %rdx, %rax       # if <=, result = eval
    ret
```

### 4.2 优势与局限

| 方面 | 说明 |
|:----|:----|
| **优势** | 避免分支预测失败，流水线友好，无分支惩罚 |
| **局限 1** | 两个分支都必须先计算（计算量大时不划算） |
| **局限 2** | 有安全风险：`p ? *p : 0` 中两个分支都会求值 |
| **局限 3** | 有副作用：`x > 0 ? x*=7 : x+=3` 不能安全使用 |

### 4.3 CMOV 指令变体

与 SET 指令相同的条件码组合：`cmove`, `cmovne`, `cmovs`, `cmovns`, `cmovg`, `cmovge`, `cmovl`, `cmovle`, `cmova`, `cmovae`, `cmovb`, `cmovbe`。

## 五、循环 (Loops)

GCC 将三种循环结构统一编译为带**向后跳转**的汇编代码。

### 5.1 Do-While 循环

最基础的循环形式，**循环体至少执行一次**。

```c
long pcount_do(unsigned long x) {
    long result = 0;
    do {
        result += x & 0x1;
        x >>= 1;
    } while (x);
    return result;
}
```

```asm
    movl    $0, %eax        # result = 0
.L2:                        # loop:
    movq    %rdi, %rdx
    andl    $1, %edx        # t = x & 0x1
    addq    %rdx, %rax      # result += t
    shrq    $1, %rdi        # x >>= 1
    jne     .L2             # if (x) goto loop
    ret
```

**通用翻译模板**：
```
loop:
    Body
    if (Test) goto loop;
```

### 5.2 While 循环

**方法一：Jump-to-Middle**
```
    goto test;
loop:
    Body
test:
    if (Test) goto loop;
```

**方法二：Guard + Do-While 转换**（GCC -O1 及以上使用）
```
    if (!Test) goto done;
loop:
    Body
    if (Test) goto loop;
done:
```

### 5.3 For 循环

`for (Init; Test; Update) Body`

先转换为 while：
```
Init;
while (Test) {
    Body
    Update;
}
```
再转换为 do-while 或 jump-to-middle 形式。

### 5.4 三种循环的汇编特征对比

| 循环类型 | 汇编特征 |
|:--------|:---------|
| **for** | 初始化 → 条件判断 → 循环体 → 更新 → 跳回判断 |
| **while** | 同 for，条件判断在开头（jump-to-middle 或 guarded-do） |
| **do-while** | 循环体至少执行一次，条件判断在末尾 |

> GCC 对 `for` 和 `while` 的优化策略基本相同，它们的汇编代码通常难以区分。

## 六、Switch 语句与跳转表 (Jump Tables)

### 6.1 跳转表原理

Switch 语句的一种高效实现是使用**跳转表**——一个存放在只读数据段中的地址数组。

```asm
jmp *.L4(,%rdi,8)     # 取 .L4 数组第 %rdi 个条目（每项 8 字节），跳转到该地址
```

### 6.2 完整示例

```asm
switcher:
    cmpq $7, %rdi
    ja    .L2               # 超出范围 -> default
    jmp   *.L4(, %rdi, 8)   # 跳转表查找

.L7:
    xorq $15, %rsi
    movq %rsi, %rdx
.L3:
    leaq  112(%rdx), %rdi
    jmp   .L6
.L5:
    leaq (%rdx, %rsi), %rdi
    salq  $2, %rdi
    jmp   .L6
.L2:
    movq  %rsi, %rdi
.L6:
    movq  %rdi, (%rcx)
    ret

.section .rodata
.L4:
    .quad   .L3      # case 0
    .quad   .L2      # case 1 -> default
    .quad   .L5      # case 2
    .quad   .L2      # case 3 -> default
    .quad   .L6      # case 4
    .quad   .L7      # case 5
    .quad   .L2      # case 6 -> default
    .quad   .L5      # case 7
```

对应的 C 代码：
```c
void switcher(long a, long b, long c, long *dest) {
    long val;
    switch (a) {
    case 5:
        c = b ^ 15;         /* fall through */
    case 0:
        val = c + 112;
        break;
    case 2:
    case 7:
        val = (c + b) << 2;
        break;
    case 4:
        val = a;
        break;
    default:
        val = b;
    }
    *dest = val;
}
```

### 6.3 Switch 编译策略对比

| 策略 | 适用场景 | 空间复杂度 | 时间复杂度 |
|:----|:---------|:----------:|:----------:|
| **跳转表** | case 值密集（范围小但 case 多） | O(n) | O(1) |
| **条件分支树** | case 值稀疏，数量少 | O(1) | O(log n) |
| **级联 if-else** | case 值极少或范围极大 | O(1) | O(n) |

### 6.4 跳转表关键点

- 存放在 `.section .rodata`（只读数据段）
- 使用 `ja`（无符号高于）检查索引是否超出范围
- 编译器自动生成默认 case 的处理（超出范围的索引指向 default）
- 跳转表占用的空间是 O(n)，但执行时间是 O(1)
- LLVM 中，case 数量 >= 4 时通常使用跳转表

## 二、x86-64 栈结构

### 2.1 栈的基本特性

- 栈地址**往低地址增长**
- `%rsp` 寄存器始终指向当前栈顶（即栈中最低的有效地址）
- 栈支持两种基本操作：`pushq` 和 `popq`

**pushq 入栈**：
```asm
pushq Rsrc
subq $8, %rsp
movq Rsrc, (%rsp)
```

**popq 出栈**：
```asm
popq Rdest
movq (%rsp), Rdest
addq $8, %rsp
```

### 2.2 栈帧结构

```
        高地址
        ┌──────────────────┐
        │ 调用者栈帧        │
        │  - 参数 7+        │  ← 若有超过 6 个参数
        ├──────────────────┤
        │ 返回地址          │  ← call 指令压入
        ├──────────────────┤
        │ 被调用者栈帧       │
        │  - 保存的寄存器    │  ← 被调用者保存的寄存器
        │  - 局部变量        │  ← 需要取地址或寄存器不够时
        │  - 参数构造区      │  ← 为下一层调用准备参数
        └──────────────────┘  ← %rsp
        低地址
```

被调用函数（callee）一定在调用函数（caller）之前返回（LIFO 特性）。

### 3.3 控制流示例

```
地址     指令
400540: push %rbx
400541: mov %rdx, %rbx
400544: callq 400550 <mult2>   # 将 0x400549 压入栈
400549: mov %rax, (%rbx)       # ← 返回地址
...

400550: mov %rdi, %rax
400553: imul %rsi, %rax
400557: retq                   # 弹出 0x400549，跳回
```

> **核心机制**：`call` 压入返回地址，`ret` 弹出返回地址，二者配合实现函数调用与返回。

## 四、参数传递约定

### 4.1 前 6 个参数（寄存器传递）

x86-64 Linux 调用约定（System V AMD64 ABI）：

| 参数序号 | 寄存器 | 说明 |
|:-------:|:------:|:-----|
| 第 1 个 | `%rdi` | 函数第一个参数 |
| 第 2 个 | `%rsi` | 函数第二个参数 |
| 第 3 个 | `%rdx` | 函数第三个参数 |
| 第 4 个 | `%rcx` | 函数第四个参数 |
| 第 5 个 | `%r8`  | 函数第五个参数 |
| 第 6 个 | `%r9`  | 函数第六个参数 |
| **返回值** | `%rax` | 函数返回值 |

### 4.2 第 7 个及之后的参数（栈传递）

- 参数 7、8... 按从**右到左**的顺序压入调用者的栈帧
- 在 x86-64 中，前 6 个整数/指针参数始终通过寄存器传递

### 4.3 浮点参数传递

- 浮点类型参数使用 XMM 寄存器：`%xmm0` ~ `%xmm7`
- 浮点返回值使用 `%xmm0`
- 所有 XMM 寄存器都是调用者保存（caller-saved）

## 五、完整的寄存器保存约定

x86-64 的全部 16 个通用寄存器及其分工：

| 寄存器 | 保存类型 | 特殊用途 |
|:------:|:--------:|:---------|
| `%rax` | **调用者保存** (Caller-saved) | 返回值 |
| `%rbx` | **被调用者保存** (Callee-saved) | — |
| `%rcx` | 调用者保存 | 第 4 个参数 |
| `%rdx` | 调用者保存 | 第 3 个参数 |
| `%rsi` | 调用者保存 | 第 2 个参数 |
| `%rdi` | 调用者保存 | 第 1 个参数 |
| `%rbp` | **被调用者保存** (Callee-saved) | 帧指针（可选） |
| `%rsp` | **被调用者保存** (Callee-saved) | 栈指针（特殊） |
| `%r8`  | 调用者保存 | 第 5 个参数 |
| `%r9`  | 调用者保存 | 第 6 个参数 |
| `%r10` | 调用者保存 | 临时数据 |
| `%r11` | 调用者保存 | 临时数据 |
| `%r12` | **被调用者保存** (Callee-saved) | — |
| `%r13` | **被调用者保存** (Callee-saved) | — |
| `%r14` | **被调用者保存** (Callee-saved) | — |
| `%r15` | **被调用者保存** (Callee-saved) | — |

### 5.1 调用者保存 (Caller-saved / Call-clobbered)

- 调用者在执行 `call` 之前，如果还需要这些寄存器的值，必须**自己保存**（通常压栈）
- 被调用者可以自由修改这些寄存器，无需恢复

### 5.2 被调用者保存 (Callee-saved / Call-preserved)

- 被调用者如果使用这些寄存器，**必须先保存旧值**，返回前恢复
- 调用者可放心使用，这些寄存器的值在 `call` 返回后保持不变

### 5.3 寄存器保护示例

```c
long call_incr2(long x) {
    long v1 = 15213;
    long v2 = incr(&v1, 3000);
    return x + v2;
}
```

```asm
call_incr2:
    pushq   %rbx                # 保存 callee-saved 寄存器
    subq    $16, %rsp
    movq    %rdi, %rbx          # 将 x 存入 %rbx（callee-saved）
    movq    $15213, 8(%rsp)
    movl    $3000, %esi
    leaq    8(%rsp), %rdi
    call    incr
    addq    %rbx, %rax          # x + v2
    addq    $16, %rsp
    popq    %rbx                # 恢复 callee-saved 寄存器
    ret
```

## 六、局部变量的分配

### 6.1 存放在寄存器中

未取地址的局部变量通常直接存储在寄存器中：

```c
long call_incr() {
    long v1 = 15213;
    long v2 = incr(&v1, 3000);
    return v1 + v2;
}
```

```asm
call_incr:
    subq    $16, %rsp           # 分配栈空间（&v1 取地址，v1 必须在栈上）
    movq    $15213, 8(%rsp)     # v1 = 15213
    movl    $3000, %esi
    leaq    8(%rsp), %rdi       # &v1
    call    incr
    addq    8(%rsp), %rax       # v1 + v2
    addq    $16, %rsp
    ret
```

### 6.2 存放在栈上

以下是局部变量必须存放在栈上的场景：

| 场景 | 原因 |
|:----|:-----|
| **取地址** (`&x`) | 寄存器没有内存地址 |
| **寄存器不够** | 局部变量超过可用寄存器数量 |
| **数组或结构体** | 需要连续内存空间 |
| **参数超过 6 个** | 第 7+ 个参数在栈上传递 |

编译器通过 `sub $N, %rsp` 在栈上分配空间，变量通过相对于 `%rsp` 的偏移访问。

## 七、尾调用优化 (Tail Call Optimization)

### 7.1 原理

当函数的最后操作是调用另一个函数并直接返回其结果时：

```c
long foo(long x) {
    return bar(x);      // 尾调用
}
```

编译器可以优化为：
```asm
foo:
    jmp bar             # 不复用栈帧，bar 的 ret 直接返回到 foo 的调用者
```

### 7.2 优化收益

| 方面 | 说明 |
|:----|:-----|
| **栈空间** | 不创建新的栈帧，复用当前栈帧 |
| **指令** | 使用 `jmp` 代替 `call`，省去压栈和弹出返回地址 |
| **递归** | 尾递归优化可将递归转化为迭代，避免栈溢出 |

> 尾调用优化在递归函数中尤为重要——尾递归被优化后空间复杂度从 O(n) 降为 O(1)。

---

## 八、递归 (Recursion)

### 8.1 递归函数的汇编实现

```c
long pcount_r(unsigned long x) {
    if (x == 0)
        return 0;
    else
        return (x & 1) + pcount_r(x >> 1);
}
```

```asm
pcount_r:
    movl    $0, %eax
    testq   %rdi, %rdi
    je      .L6
    pushq   %rbx                # 保护 callee-saved 寄存器
    movq    %rdi, %rbx
    andl    $1, %ebx            # %ebx = x & 1
    shrq    %rdi                # x >>= 1
    call    pcount_r            # 递归调用
    addq    %rbx, %rax          # (x&1) + 返回值
    popq    %rbx                # 恢复 callee-saved 寄存器
.L6:
    ret
```

### 8.2 递归的关键支持机制

- 每个函数调用实例都有一个**私有的栈帧**
- 栈帧保存寄存器、局部变量和返回地址
- 寄存器保存约定避免不同调用层之间的数据覆盖
- 栈的 LIFO 特性完美匹配函数调用的嵌套模式
- 栈空间耗尽会导致**栈溢出**（Segmentation Fault）

### 8.3 递归与迭代的选择

| 方面 | 递归 | 迭代 |
|:----|:----|:-----|
| **栈空间** | O(n)，每层一个新栈帧 | O(1)，复用同一栈帧 |
| **可读性** | 对分治/树形问题更自然 | 对线性问题更直接 |
| **尾递归优化后** | 空间 O(1)，性能等同迭代 | — |

---

## 九、何时使用栈帧

编译器会在以下情况下创建栈帧（`sub $N, %rsp`）：

1. **取地址操作**：局部变量被 `&` 引用
2. **参数过多**：参数数量超过 6 个
3. **局部变量过多**：可用寄存器不够分配
4. **数组或结构体**：需要连续内存/对齐
5. **调用子函数**：需要保存调用者状态

## 总结

| 概念 | 要点 |
|:----|:------|
| **call/ret** | call 压入返回地址，ret 弹出并跳转 |
| **参数传递** | 前 6 个用寄存器（%rdi/%rsi/%rdx/%rcx/%r8/%r9），7+ 用栈 |
| **返回值** | 通过 %rax 传递 |
| **寄存器约定** | 调用者保存（caller-saved）vs 被调用者保存（callee-saved） |
| **局部变量** | 未取地址用寄存器，取地址/数组/结构体用栈 |
| **尾调用优化** | jmp 代替 call，复用栈帧，节省空间 |
| **递归** | 每次调用创建新栈帧，栈的 LIFO 特性天然支持 |

## 一、局部变量的存储

### 1.1 两种存储方式

**方式一：寄存器（未取地址的局部变量）**

```c
int returnOne(void) {
```

```c
int returnOneTwo(void) {
```

## 二、数组 (Arrays)

### 2.1 一维数组

**原理**：`T A[L]` 在内存中分配连续的 `L * sizeof(T)` 字节。

```c
char string[12];    // 12 bytes
int val[5];         // 20 bytes (4*5)
double a[3];        // 24 bytes (8*3)
char *p[3];         // 24 bytes (8*3)
```

**数组的访问**：A 可以视为指向数组第一个元素的常量指针 `T*`。

```c
int val[5] = {1, 5, 2, 1, 3};
// val[4]     → int,    值为 3
// val        → int*,   值为 x
// val+1      → int*,   值为 x+4
// &val[2]    → int*,   值为 x+8
// *(val+1)   → int,    值为 5
```

### 2.2 数组步长 (Stride)

在汇编层，数组访问通过**基址 + 偏移量 × 比例因子**实现，步长编码在指令中：

```asm
movl (%rdi, %rsi, 4), %eax    # 取 arr[index]，步长 4 编码在指令中
```

| 数组类型 | 元素大小 | 步长 |
|:--------|:--------:|:----:|
| `int[]` | 4 字节 | 4 |
| `char[]` | 1 字节 | 1 |
| `long[]` | 8 字节 | 8 |
| `short[]` | 2 字节 | 2 |

### 2.3 嵌套二维数组 (Nested Arrays)

**原理**：`T A[R][C]` — R 行 C 列，**行优先 (Row-Major)** 布局。

- 内存大小：`R × C × K` 字节 (K = sizeof(T))
- 元素 `A[i][j]` 地址：`A + (i × C + j) × K`

```
A[0][0] A[0][1] ... A[0][C-1] A[1][0] A[1][1] ... A[R-1][C-1]
<──── 第0行 ────><──── 第1行 ────> ... <─── 第R-1行 ────>
```

**元素访问示例**：
```c
int get_pgh_digit(int index, int dig) {
    return pgh[index][dig];
}
```

```asm
leaq  (%rdi, %rdi, 4), %rax    # 5*index
addl  %rax, %rsi               # 5*index + dig
movl  pgh(, %rsi, 4), %eax     # M[pgh + 4*(5*index + dig)]
```

**定长数组（编译时已知维度）**：
```asm
salq $6, %rsi            # 64*i (16*4=64, 16 是列数)
addq %rsi, %rdi          # a + 64*i
movl (%rdi, %rdx, 4), %eax  # M[a + 64*i + 4*j]
```

**变长数组（运行时维度）**：
```asm
imulq %rdx, %rdi         # n*i (需要整数乘法，因为 n 未知)
leaq  (%rsi, %rdi, 4), %rax  # a + 4*n*i
movl  (%rax, %rcx, 4), %eax  # a + 4*n*i + 4*j
```

### 2.4 多级数组 (Multilevel Arrays)

```c
int8_t first[3]  = {0x00, 0x01, 0x02};
int8_t second[3] = {0x10, 0x11, 0x12};
int8_t *multilevel[2] = {first, second};
```

- 外层数组存储的是**指针**（每个 8 字节）
- 内层数组可以独立分配在内存的不同位置
- 访问需要**两次内存访问**（一次取指针，一次取数据）

### 2.5 两种方式的对比

| 特性 | 嵌套数组 | 多级数组 |
|:----|:---------|:---------|
| **内存布局** | 连续的行优先 | 指针数组 + 独立行 |
| **访问方式** | `Mem[A + C*i + j]` | `Mem[Mem[A + 8*i] + 4*j]` |
| **内存访问次数** | 1 次 | 2 次 |
| **步长** | 编译时固定 | 运行时确定 |
| **灵活性** | 每行长度必须相同 | 每行长度可以不同 |
| **使用场景** | 固定大小矩阵 | 不规则结构（如字符串数组） |

## 三、结构体 (Structs)

### 3.1 内存分配

结构体分配一块连续内存，大小足够装下所有成员变量，按声明顺序分配。

```c
struct rec {
    int a[4];           // offset: 0,  size: 16
    size_t i;           // offset: 16, size: 8
    struct rec *next;   // offset: 24, size: 8
};
// Total: 32 bytes
```

```
a[0..3]       i        next
├── 16B ──┤ ├── 8B ──┤ ├── 8B ──┤
0         16        24        32
```

### 3.2 对齐规则 (Alignment)

**基本原则**：
1. 成员变量的类型占用 K 字节，则该成员变量的偏移必须是 K 的倍数（**自然对齐**）
2. 结构体的对齐需求为 Kmax（各成员对齐需求的最大值）
3. 结构体的**起始地址**和**总大小**必须是 Kmax 的倍数

### 3.3 x86-64 基本类型对齐要求

| 类型 | 对齐要求 |
|:----|:---------|
| `char` | 1 字节（无限制） |
| `short` | 2 字节（地址最低 1 位为 0） |
| `int`, `float` | 4 字节（地址最低 2 位为 00） |
| `double`, `long`, `char*` | 8 字节（地址最低 3 位为 000） |
| `long double` (Linux GCC) | 16 字节 |

### 3.4 对齐示例与填充 (Padding)

```c
struct S1 {
    char c;        // offset: 0,  size: 1
    // 3 bytes padding ← 为 int 对齐
    int i[2];      // offset: 4,  size: 8
    double v;      // offset: 16, size: 8
};
// Total: 24 bytes, alignment: 8
```

```
c │ padding │  i[0..1]  │     v     │
1B    3B       8B          8B
0    4        8          16        24
```

### 3.5 字段重排对大小的影响

按对齐要求**从大到小**排列成员可以最小化填充：

```c
// 较差的布局：24 bytes
struct S4 {
    char c;        // +1
    // +3 padding
    int i;         // +4
    char d;        // +1
    // +3 padding
};

// 更好的布局：12 bytes（节省 50%）
struct S5 {
    int i;         // +4
    char c;        // +1
    char d;        // +1
    // +2 padding
};
```

**经验法则**：将结构体成员按**对齐要求从大到小**排列，可以最小化填充，减少内存浪费。

### 3.6 为什么需要对齐

- 内存访问以字长为基本单位（32 或 64 位）
- **不对齐**的变量需要多次内存访问（可能跨 cache line 边界）
- 编译器在成员之间和结构体末尾插入填充字节来确保对齐
- 一些架构（如早期的 ARM）直接禁止未对齐访问

## 五、联合体 (Union)

联合体由**最大的成员变量**决定大小，所有成员共享同一块内存区域：

```c
union U1 {
    char c;        // 1 byte
    int i[2];      // 8 bytes
    double v;      // 8 bytes
};
// Total: 8 bytes

typedef union {
    float f;
    unsigned u;
} bit_float_t;

float bit2float(unsigned u) {
    bit_float_t arg;
    arg.u = u;
    return arg.f;  // 保留位串不变，按 float 重新解释
}
```

> Union 常用于位串的重新解释、类型双关（type punning）以及节省内存的场景。

## 六、字节序 (Endianness)

### 6.1 Little-Endian（x86-64 采用）

**低地址存放最低有效字节**（LSB first）。

```c
int courses[4] = {0x15213, ...};
// 十六进制值：0x00015213
// 内存中的字节顺序（小端）：
// 地址递增 → 13 52 01 00
```

### 6.2 Big-Endian

**低地址存放最高有效字节**（MSB first）。

大端模式下，同一 `0x00015213` 的内存布局为：`00 01 52 13`。

### 6.3 小端序的特点

| 方面 | 说明 |
|:----|:------|
| **直观性** | 人类直接阅读不便（字节顺序"颠倒"） |
| **类型转换便利性** | 窄化转换（如 `long*` → `int*`）不需要移动指针，低字节自然对齐 |
| **地址运算** | `(int*)p` 和 `p` 指向同一地址，低字节在低地址 |
| **网络字节序** | 网络协议通常使用大端，需用 `htonl`/`ntohl` 转换 |

## 七、指针与数组的重要辨析

| 声明 | `A` 类型 | `sizeof(A)` | 说明 |
|:----|:---------|:-----------:|:-----|
| `int A1[3]` | `int[3]`（数组） | 12 | 数组名是首元素地址 |
| `int *A2` | `int*`（指针） | 8 | 指针本身占 8 字节 |
| `int A1[3][5]` | `int[3][5]` | 60 | 二维数组连续存储 |
| `int *A2[3]` | `int*[3]` | 24 | 指针数组 |
| `int (*A3)[3]` | `int(*)[3]` | 8 | 数组指针 |

> 关键区别：`A1` 是数组类型，`A2` 是指针类型。在大多数表达式中，数组名退化为指向首元素的指针，但在 `sizeof` 和 `&` 运算符下保持数组类型。

## 八、浮点运算补充

### 8.1 浮点指令集演进

| 指令集 | 说明 |
|:------|:------|
| x87 | 过时的浮点指令集 |
| SSE (SSE2/3/4) | 流式 SIMD 扩展，支持向量指令 |
| AVX/AVX2/AVX-512 | 高级向量扩展，最新 x86-64 扩展 |

### 8.2 XMM 寄存器

16 个 128 位寄存器（`%xmm0` ~ `%xmm15`），可看作：
- 16 个单字节整数 / 8 个 16 位整数 / 4 个 32 位整数
- 4 个单精度浮点数 / 2 个双精度浮点数

### 8.3 浮点运算示例

```c
float fadd(float x, float y) { return x + y; }
double dadd(double x, double y) { return x + y; }
```

```asm
addss   %xmm1, %xmm0      # 单精度加 (scalar single)
ret

addsd   %xmm1, %xmm0      # 双精度加 (scalar double)
ret
```

### 8.4 浮点参数传递约定

- 整数/指针参数：通用寄存器 (`%rdi`, `%rsi`, ...)
- 浮点参数：XMM 寄存器 (`%xmm0`, `%xmm1`, ...)
- 浮点返回值：`%xmm0`
- 所有 XMM 寄存器都是**调用者保存** (caller-saved)
- 浮点比较指令 `ucomiss`/`ucomisd` 设置 `CF`、`ZF`、`PF`（遇到 NaN 时行为特殊）

## 一、缓冲区溢出原理

### 1.1 什么是缓冲区溢出

缓冲区溢出（Buffer Overflow）是指**写入超出数组分配边界的操作**，也称为**栈破坏（Stack Smashing）**。这是导致计算机安全漏洞最重要的技术原因之一。

### 1.2 危险函数：gets()

```c
char *gets(char *dest) {
    int c = getchar();
    char *p = dest;
    while (c != EOF && c != '\n') {
        *p++ = c;
        c = getchar();
    }
    *p = '\0';
    return dest;
}
```

**核心问题**：
- `gets()` 不知道目标缓冲区 `dest` 有多大
- 只有在遇到 `EOF` 或 `\n` 时才停止读取
- 输入超过缓冲区大小时，会发生溢出，覆盖栈上的其他数据

> `gets()` 已被 C11 标准废弃，**永远不要在生产代码中使用**。

### 1.3 其他危险函数

| 危险函数 | 问题 | 安全替代 |
|:--------|:-----|:---------|
| `gets` | 无边界检查 | `fgets(buf, size, stdin)` |
| `strcpy` | 无边界检查 | `strncpy` |
| `strcat` | 无边界检查 | `strncat` |
| `sprintf` | 无边界检查 | `snprintf` |
| `scanf("%s")` | 无边界检查 | `fgets` 或 `%ns` 格式 |

## 二、栈溢出攻击原理

### 2.1 易受攻击的代码

```c
void echo() {
    char buf[4];              // 数组太小！
    gets(buf);                // 如果输入 > 4 会怎么样？
    puts(buf);
}
```

反汇编（编译器实际分配 24 字节）：
```asm
echo:
    subq    $24, %rsp         # 分配栈空间
    movq    %rsp, %rdi
    callq   gets
    movq    %rsp, %rdi
    callq   puts
    addq    $24, %rsp
    ret
```

### 2.2 栈布局

```
        高地址
        +------------------+  初始 %rsp
+0x28   | 调用者的栈帧      |
+0x20   | 返回地址          |  ← 溢出后可被覆盖
+0x18   | 未使用空间        |
        |  (20 bytes)       |
+0x00   | buf[0..3]         |  ← %rsp
        +------------------+
        低地址
```

### 2.3 四种溢出情形

| 情形 | 描述 | 程序行为 |
|:----|:-----|:---------|
| **1. 无破坏** | 溢出但未触及返回地址 | 正常执行 |
| **2. 段错误** | 返回地址被覆盖为无效地址 | Segmentation Fault |
| **3. 巧合执行** | 返回地址被修改为合法地址 | 看似正常但行为异常 |
| **4. 代码注入** | 返回地址指向注入的恶意代码 | **被攻击** |

### 2.4 代码注入攻击

攻击输入格式：
```
[恶意代码字节] [填充] [指向恶意代码的返回地址]
```

1. 恶意代码（shellcode）放置在栈缓冲区中
2. 填充使溢出精确覆盖返回地址
3. 返回地址指向缓冲区的起始位置
4. `ret` 执行时跳转到恶意代码

攻击示例（简单的汇编代码注入）：
```asm
mov  $0xdecafbad, %eax    # 机器码：b8 ad fb ca de
ret                        # 机器码：c3
```

输入字符串布局：
```
[缓冲区 24 字节填充] [指向缓冲区的地址] [shellcode]
```

### 2.5 缓冲区溢出攻击历史事件

| 事件 | 年份 | 说明 |
|:----|:----:|:-----|
| **Internet Worm** | 1988 | 利用 `fingerd` 的 `gets()` 漏洞，几小时内感染约 6000 台电脑（当时互联网的 10%） |
| **IM Wars** | 1999 | AOL 利用 AIM 客户端中的缓冲区溢出漏洞识别和阻止 MSN 客户端 |
| **Wii Twilight Hack** | 2000s | 利用缓冲区溢出在游戏机上运行自制软件 |

## 三、防御机制

### 3.1 防御一：安全编程

程序员使用安全的库函数替代危险函数：

| 危险函数 | 安全替代 | 说明 |
|:--------|:---------|:-----|
| `gets(buf)` | `fgets(buf, size, stdin)` | `size` 指定最大读取长度 |
| `strcpy` | `strncpy` | 指定最大复制长度 |
| `strcat` | `strncat` | 指定最大连接长度 |
| `sprintf` | `snprintf` | 指定输出缓冲区大小 |
| `scanf("%s")` | `fgets` + `sscanf` | 先限长再解析 |

```c
char *fgets(char *s, int size, FILE *stream);
// size 参数指定最大读取长度，即使输入过长也不会溢出缓冲区
```

> 这是**最有效的防御**——从源头上消除漏洞。永远不要假设用户输入是安全的。

### 3.2 防御二：栈随机化 (ASLR)

**地址空间布局随机化 (Address Space Layout Randomization)**：

- 操作系统将栈基址随机化，每次程序运行栈的起始地址不同
- 攻击者难以预测注入代码的地址
- 综合 ASLR 和 NX 机制已广泛部署于 Linux、Windows、macOS

```
main
┌────────────────────┐
│    Application     │
├────────────────────┤
│  Random Allocation │ ← 每次运行地址不同
├────────────────────┤
│  Stack             │ ← 缓冲区地址每次不同
└────────────────────┘
```

**局限性**：暴力枚举仍然可能成功，尤其是在 32 位系统中（地址空间较小）。

### 3.3 防御三：栈不可执行 (NX-bit / W^X)

x86-64 增加了内存区域的**可执行权限标记**：

| MMU 权限位 | 作用 |
|:----------:|:----:|
| SUP | 管理权限 |
| READ | 只读权限 |
| WRITE | 可写权限 |
| EXEC | 可执行权限 |

- 栈区域标记为**不可执行**（NX-bit）
- 堆区域同样标记为不可执行
- 尝试运行栈上的代码会触发段错误
- 有效防御**代码注入攻击**，但无法防御 ROP

### 3.4 防御四：栈金丝雀 (Stack Canary)

**思路**：在栈上缓冲区后面插入一个特殊值（金丝雀），函数返回前检测该值是否被破坏。

**GCC 选项**：`-fstack-protector`（默认开启）

```asm
echo_with_canary:
    subq    $24, %rsp
    movq    %fs:0x28, %rax      # * 从线程本地存储加载 canary
    movq    %rax, 8(%rsp)       # * 将 canary 存入栈上（返回地址前）
    xorl    %eax, %eax
    movq    %rsp, %rdi
    callq   gets
    movq    %rsp, %rdi
    callq   puts
    movq    8(%rsp), %rax       # * 从栈上取出 canary
    xorq    %fs:0x28, %rax      # * 与原始 canary 比较
    je      .L6                 #   If same, OK
    callq   __stack_chk_fail    #   FAIL: 栈破坏检测
.L6:
    addq    $24, %rsp
    ret
```

**Canary 的完整校验逻辑（伪代码）**：
```
canary = *(%fs:0x28)                 # 从线程本地存储获取
stack[canary_offset] = canary        # 存入栈上（缓冲区与返回地址之间）
gets(buffer)                         # 有风险的调用
if stack[canary_offset] != canary:   # 函数返回前校验
    __stack_chk_fail()               # 栈被破坏，终止程序
```

**Canary 的栈布局**：
```
调用 gets() 之前的栈：
┌──────────────────────┐
│      返回地址        │
├──────────────────────┤
│   金丝雀值 (8 bytes) │  ← 返回地址前
├──────────────────────┤
│   未使用空间         │
├──────────────────────┤ ← %rsp
│ buf[3..0]            │
└──────────────────────┘
```

**Canary 特性**：

| 特性 | 说明 |
|:----|:------|
| **存储位置** | 线程本地存储，`%fs:0x28`（Linux） |
| **检查时机** | 函数返回前 |
| **溢出检测** | 缓冲区溢出会覆盖 canary，导致检查失败 |
| **失败处理** | `__stack_chk_fail()` 终止程序，报告 "stack smashing detected" |
| **名称来源** | 煤矿工人用金丝雀检测有毒气体——金丝雀比人类更敏感 |

**Canary 的成本**：
- **内存**：多占用栈上 8 字节
- **时间**：每次调用增加 3 条额外指令（加载、存储、检查）
- **局限性**：无法防御非连续内存访问（如通过指针间接覆盖返回地址）

## 四、ROP 攻击 (Return-Oriented Programming)

### 4.1 背景

当 NX-bit（栈不可执行）启用后，攻击者无法直接在栈上执行代码。ROP 作为一种**代码复用攻击**应运而生。

### 4.2 核心思想

1. **不再注入新代码**，而是从已有的可执行代码（如 libc）中寻找以 `ret` 结尾的指令片段
2. 这些片段称为 **gadget**（工具链）
3. 将多个 gadget 的地址串联在栈上
4. 每次 `ret` 执行一个 gadget，链式完成攻击

**`ret` 指令的机器码**：`0xc3`

### 4.3 Gadget 示例

从现有代码中搜索字节序列：

```asm
4004d0: 48 0f af fe    imul %rsi,%rdi
4004d4: 48 8d 04 17    lea  (%rdi,%rdx,1),%rax   # 可作为 gadget
4004d8: c3             retq
```

更精巧的方式——利用指令的**非对齐边界**：

```asm
<setval>:
   4004d9: c7 07 d4 48 89 c7    movl $0xc78948d4, (%rdi)
   4004df: c3                   retq

   4004dc: 48 89 c7             movq %rax, %rdi    # 另一个 Gadget!
```

### 4.4 ROP 攻击的栈布局

```
           P's Stack Frame
           ┌──────────────────┐
           │  Gadget 1 地址   │ ← ret → 跳转到 gadget1
           ├──────────────────┤
           │  Gadget 2 地址   │ ← gadget1 的 ret → 跳转到 gadget2
           ├──────────────────┤
           │  Gadget 3 地址   │ ← gadget2 的 ret → 跳转到 gadget3
           ├──────────────────┤
           │  ...             │
           ├──────────────────┤ ← gets() 覆盖的区域
           │      pad         │
           │      pad         │
           ├──────────────────┤
           │      buf         │
           └──────────────────┘
```

### 4.5 ROP 与各防御机制的关系

| 防御机制 | 能否防御 ROP | 原因 |
|:---------|:-----------:|:-----|
| **ASLR** | 部分 | 代码基地址仍可预测或泄漏 |
| **NX-bit** | **不能** | 代码在已有可执行段中，不在栈上 |
| **Stack Canary** | 能 | 溢出覆盖 canary 时会被检测到 |
| **安全编程 (fgets)** | 能 | 从根源上阻止溢出 |

### 4.6 对 ROP 的进一步防御

| 防御技术 | 说明 |
|:---------|:-----|
| **控制流完整性 (CFI)** | 在间接跳转/调用前检查目标地址是否在合法集合中 |
| **影子栈 (Shadow Stack)** | 硬件维护独立的返回地址副本（CET） |
| **地址空间布局随机化增强** | 代码段也随机化，增加 gadget 定位难度 |

## 总结

| 攻击/防御 | 原理 | 应对措施 |
|:---------|:-----|:---------|
| **缓冲区溢出** | gets() 不限制输入长度，覆盖栈上返回地址 | 使用 fgets() 等安全函数 |
| **代码注入攻击** | 返回地址指向栈上的注入代码 | NX-bit（栈不可执行） |

| **Stack Canary** | 在返回地址前插入检查值，溢出后检测 | `__stack_chk_fail` 终止程序 |
| **ROP** | 利用现有代码中的 gadget 链绕过 NX-bit | 控制流完整性 (CFI)、影子栈 (CET) |

> **安全原则**：永远不要假设用户输入是安全的。使用边界检查的函数，启用编译器和操作系统提供的所有防御机制。

## 一、为什么使用 GDB

GDB（GNU Debugger）是 Linux/Unix 环境下最广泛使用的调试器，主要用于：

- 查看程序运行时的内部状态
- 跟踪程序的执行路径
- 分析崩溃时的调用栈和变量值
- 反汇编并单步执行机器指令
- 调试多进程、多线程程序

### 1.1 被调试文件类型

| 类型 | 说明 |
|:----|:------|
| **普通可执行文件** | 可通过 `gcc -g` 生成调试符号 |
| **不带调试符号的可执行文件** | 仍可调试，但看不到源码和变量名 |
| **运行中的进程** | 通过 `--pid` 或 `attach` 附加 |
| **Core dump** | 程序崩溃后生成的内存快照 |

### 1.2 编译带调试信息的程序

```bash
gcc -g -o program source.c        # 编译时加入调试符号（-g）
gcc -g -O0 -o program source.c    # -O0 禁用优化，调试更准确
gcc -g -O1 -o program source.c    # -O1 轻度优化，部分变量可能被优化掉
```

> **注意**：编译器优化（-O1 及以上）可能使 GDB 显示的行号、变量值不准确。调试阶段建议使用 `-O0`。

## 三、断点 (Breakpoints)

### 3.1 设置断点

```gdb
break main                  # 在 main 函数入口
break 42                    # 在当前文件第 42 行
break file.c:15             # 在 file.c 第 15 行
break file.c:main           # 在 file.c 的 main 函数
break *0x400544             # 在内存地址 0x400544
break main if x == 5        # 条件断点：x 等于 5 时触发
break location thread 3     # 只在线程 3 中触发
```

### 3.2 条件断点详解

```gdb
break func if err != 0      # 出错时触发
break loop if i > 100       # 循环 100 次后触发
condition 2 i > 1000        # 为编号 2 的断点添加条件
```

### 3.3 断点管理

```gdb
info breakpoints        # 查看所有断点（显示编号和状态）
delete 2                # 删除编号 2 的断点
delete                  # 删除所有断点
clear                   # 删除当前行的断点
clear func              # 删除指定函数的断点
disable 2               # 禁用编号 2 的断点（不删除）
enable 2                # 重新启用编号 2 的断点
```

> **断点 vs 监视点**：断点在代码位置停止，监视点在数据变化时停止。

## 四、运行控制

### 4.1 基本命令


### 4.2 逐级执行的区别

## 一、Linux 文件系统

### 1.1 基本概念

- Linux 没有盘符（C:/D:）的概念，所有存储设备挂载在根目录 `/` 下
- 文件系统使用**目录树**结构，从根目录 `/` 开始
- 路径分隔符为 `/`（Windows 使用 `\`）
- Linux 不使用 NTFS、exFAT 或 APFS

### 1.2 常用目录标识

| 符号 | 含义 |
|:----:|:------|
| `~` | 当前用户的家目录（`/home/用户名`） |
| `.` | 当前目录 |
| `..` | 上级目录 |
| `/` | 根目录 / 路径分隔符 |

### 1.3 文件系统层级结构 (FHS)

| 目录 | 用途 |
|:----|:------|
| `/bin` | 基本用户命令（如 `ls`, `cp`, `mv`） |
| `/sbin` | 系统管理命令 |
| `/etc` | 系统配置文件 |
| `/home` | 用户家目录 |
| `/tmp` | 临时文件 |
| `/var` | 可变数据（日志、缓存等） |
| `/usr` | 用户程序和数据 |
| `/proc` | 虚拟文件系统（进程和内核信息） |
| `/dev` | 设备文件 |

### 1.4 Linux 下的文件特点

- **换行符**：Linux/macOS 使用 LF (`\n`)，Windows 使用 CRLF (`\r\n`)
- **编码**：通常使用 UTF-8
- **文件类型**：使用 `file` 命令判断
- **无回收站**：`rm` 删除后不可恢复
- **执行当前目录程序**：需使用 `./program`

## 三、GCC 编译

### 3.1 编译流程

GCC 将 C 源代码编译为可执行文件，经历四个阶段：

```
source.c → [预处理] → source.i → [编译] → source.s → [汇编] → source.o → [链接] → program
```

| 阶段 | 命令 | 输出 |
|:----|:-----|:-----|
| 预处理 | `gcc -E source.c -o source.i` | 展开宏、处理 `#include` |
| 编译 | `gcc -S source.i -o source.s` | 生成汇编代码 |
| 汇编 | `gcc -c source.s -o source.o` | 生成目标文件（机器码） |
| 链接 | `gcc source.o -o program` | 生成可执行文件 |

### 3.2 常用编译选项

```bash
gcc -o program source.c                     # 编译链接一步完成

gcc -g -o program source.c                  # 加入调试符号（GDB 调试必备）
gcc -O0 -o program source.c                 # 不优化
gcc -O1 -o program source.c                 # 轻度优化
gcc -O2 -o program source.c                 # 中度优化（常用）
gcc -O3 -o program source.c                 # 激进优化

gcc -c file1.c -o file1.o                   # 编译但不链接
gcc -c file2.c -o file2.o
gcc file1.o file2.o -o program              # 链接目标文件

gcc -Wall source.c -o program               # 开启所有常见警告
gcc -Wall -Wextra source.c -o program       # 更多警告
gcc -I./include source.c -o program         # 添加头文件搜索路径
gcc -L./lib source.c -lmylib -o program     # 添加库搜索路径并链接
gcc -fno-stack-protector source.c           # 禁用栈保护（canary）
gcc -S source.c -o source.s                 # 只生成汇编代码
gcc -E source.c -o source.i                 # 只进行预处理
```

### 3.3 常用编译选项速查

| 选项 | 说明 |
|:----|:------|
| `-o <file>` | 指定输出文件名 |
| `-g` | 生成调试符号 |
| `-O0` ~ `-O3` | 优化级别 |
| `-Wall` | 开启常见警告 |
| `-Wextra` | 开启额外警告 |
| `-Werror` | 将警告视为错误 |
| `-I <dir>` | 添加头文件搜索目录 |
| `-L <dir>` | 添加库文件搜索目录 |
| `-l<name>` | 链接指定库（如 `-lm` 链接数学库）|
| `-c` | 只编译不链接 |
| `-S` | 只生成汇编代码 |
| `-E` | 只进行预处理 |
| `-std=c99` / `-std=c11` | 指定 C 语言标准 |
| `-fno-stack-protector` | 禁用栈金丝雀保护 |
| `-fstack-protector` | 启用栈金丝雀保护（默认） |
| `-m32` / `-m64` | 生成 32/64 位代码 |

## 四、Makefile 基础

### 4.1 Makefile 结构

Makefile 定义了一系列文件的编译方式及其依赖关系：

```makefile
target: prerequisites
	recipe
```

- **target**：要生成的文件名
- **prerequisites**：生成 target 所依赖的文件
- **recipe**：生成 target 的命令（**必须以 Tab 缩进**）

### 4.2 示例：单个文件的 Makefile

```makefile
CC = gcc
CFLAGS = -Wall -g -O0

program: program.c
	$(CC) $(CFLAGS) -o program program.c

clean:
	rm -f program
```

### 4.3 示例：多文件的 Makefile

```makefile
CC = gcc
CFLAGS = -Wall -g -O0
OBJS = main.o utils.o io.o

program: $(OBJS)
	$(CC) $(CFLAGS) -o program $(OBJS)

main.o: main.c header.h
	$(CC) $(CFLAGS) -c main.c

utils.o: utils.c header.h
	$(CC) $(CFLAGS) -c utils.c

io.o: io.c header.h
	$(CC) $(CFLAGS) -c io.c

.PHONY: clean
clean:
	rm -f $(OBJS) program
```

### 4.4 简化写法：使用自动变量

```makefile
CC = gcc
CFLAGS = -Wall -g -O0

program: main.o utils.o io.o
	$(CC) $(CFLAGS) -o $@ $^

%.o: %.c header.h
	$(CC) $(CFLAGS) -c $< -o $@

clean:
	rm -f *.o program
```

| 自动变量 | 含义 |
|:--------:|:------|
| `$@` | 目标文件名 |
| `$^` | 所有依赖文件（去重） |
| `$<` | 第一个依赖文件 |
| `$?` | 比目标更新的依赖文件 |

## 五、用户与权限

### 5.1 Root 用户

- root 是 Linux 具有最高权限的用户
- 使用 `sudo <command>` 以 root 权限执行命令
- 使用 `su` 切换用户（`su - username`）
- 被授权的用户可通过 `/etc/sudoers` 配置 `sudo` 权限

### 5.2 文件权限

```bash
ls -l                    # 查看文件权限
-rwxr-xr-x 1 user group 4096 Jun 15 10:00 program
```

权限表示（10 个字符）：

| 位置 | 含义 |
|:----:|:------|
| 第 1 位 | 文件类型（`-` 普通文件，`d` 目录，`l` 链接） |
| 第 2-4 位 | **属主**权限（rwx） |
| 第 5-7 位 | **属组**权限（rwx） |
| 第 8-10 位 | **其他用户**权限（rwx） |

| 权限 | 二进制 | 数字 | 含义 |
|:----:|:-----:|:----:|:------|
| `---` | 000 | 0 | 无权限 |
| `--x` | 001 | 1 | 仅可执行 |
| `-w-` | 010 | 2 | 仅可写 |
| `-wx` | 011 | 3 | 可写可执行 |
| `r--` | 100 | 4 | 仅可读 |
| `r-x` | 101 | 5 | 可读可执行 |
| `rw-` | 110 | 6 | 可读可写 |
| `rwx` | 111 | 7 | 全部权限 |

### 5.3 chmod 和 chown

```bash
chmod u+x file          # 属主增加执行权限
chmod g-w file          # 属组移除写权限
chmod o=r file          # 其他用户设为只读
chmod a+x file          # 所有用户增加执行权限

chmod 755 file          # rwxr-xr-x（属主全部，其他只读执行）
chmod 644 file          # rw-r--r--（属主读写，其他只读）
chmod 600 file          # rw-------（仅属主读写）

chmod -R 755 dir/       # 递归修改目录权限

chown user file         # 更改文件属主
chown user:group file   # 同时更改属主和属组
chown -R user:group dir/ # 递归更改
```

## 六、I/O 重定向与管道

### 6.1 重定向

```bash

command 2> file         # 将错误输出写入文件

command &> file         # 同上（bash 简写）
command >> file 2>&1    # 追加模式

```

### 6.2 管道

```bash
cat file | grep "error" | head -20  # 搜索前 20 个包含 "error" 的行
dmesg | tail -20                # 查看最近 20 条内核消息
```

### 6.3 管道与重定向组合

